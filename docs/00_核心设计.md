## 一、系统概述

### 1.1 本系统解决什么问题

个人成长面临三大挑战：

| 挑战 | 表现 | 传统解决方案的局限 |
|-----|------|------------------|
| **方向迷失** | 忙了一年，回头发现没有实质进展 | OKR/年度计划容易流于形式，缺乏持续校准机制 |
| **执行低效** | 项目拖延、任务堆积、优先级混乱 | 待办清单越来越长，但完成率越来越低 |
| **经验流失** | 同样的坑踩两次，学过的东西记不住 | 笔记软件堆满资料，但真正能复用的很少 |

**PersonalOS 的解决思路**：

```
方向迷失 → 战略层校准机制（月度/季度战略对话）
执行低效 → 项目层管理机制（周度复盘+AI项目经理）
经验流失 → 能力层复利机制（日度沉淀+知识网络）
```

### 1.2 核心理念

**1. 人机协作：AI 是合伙人，不是工具**

传统 AI 使用方式是「你问我答」，PersonalOS 的方式是「共同协作」：
- AI 有自己的工作空间，可以自主探索
- AI 有记忆系统，越用越懂你
- AI 有明确的角色分工，不只是通用助手

**2. 复利工程：越用越强，经验可沉淀**

每次任务结束不是终点，而是下次的起点：
- 规则库：什么时候该做什么（决策指南）
- 技能库：如何做某件事（方法积累）
- 记忆系统：近期动态 + 长期稳定

**3. 战略驱动：行动对齐长期目标**

不是「有什么做什么」，而是「该做什么才做什么」：
- 北极星指标：长期方向
- 季度目标：阶段重点
- 周度复盘：持续校准

### 1.3 系统的两大模块

PersonalOS 由两大模块组成：

```
┌─────────────────────────────────────────────────────┐
│                   PersonalOS                         │
│                                                      │
│  ┌─────────────────┐    ┌─────────────────────────┐ │
│  │  人机协作模块    │    │     成长管理模块         │ │
│  │                 │    │                         │ │
│  │ • 四区工作空间   │    │ • 战略层（方向校准）     │ │
│  │ • 权限边界      │    │ • 项目层（目标推进）     │ │
│  │ • 复利机制      │    │ • 能力层（知识积累）     │ │
│  │ • AI角色分工    │    │ • 三层反馈循环          │ │
│  │                 │    │                         │ │
│  │ 解决：如何协作   │    │ 解决：如何成长          │ │
│  └─────────────────┘    └─────────────────────────┘ │
│                                                      │
│            两个模块相互支撑，缺一不可                  │
└─────────────────────────────────────────────────────┘
```

**为什么需要两个模块？**

- 只有人机协作模块 → 有强大的 AI 助手，但缺乏成长方向，容易「瞎忙」
- 只有成长管理模块 → 有清晰的成长框架，但 AI 角色模糊，协作低效

两个模块结合：
- 人机协作模块定义「如何高效协作」
- 成长管理模块定义「协作去做什么」

## 二、系统架构

### 2.1 四区工作空间（顶层设计）

```
PersonalOS/
│
├── 00_Core/        # 游戏规则（人主导，AI遵守）
│
├── 01_Shared/      # 协作区（PDCA系统，AI可读）
│
├── 02_Human/       # 私人空间（AI不进入）
│
└── 03_AI/          # AI自主空间（AI管理，人观察）
```

**四区的定位**：

| 区域 | 定位 | 谁主导 | AI权限 |
|-----|------|-------|--------|
| **00_Core/** | 系统的「宪法」，定义协作规则 | 人 | 可读，不可改 |
| **01_Shared/** | 协作的「主战场」，PDCA系统运行于此 | 双方 | 可读可写 |
| **02_Human/** | 人的「私人空间」，AI不主动进入 | 人 | 需授权才能读 |
| **03_AI/** | AI的「自主空间」，AI自由探索 | AI | 完全自主 |

### 2.5 权限边界详细规则

**00_Core/ 权限规则**：

```
AI可以：
✅ 随时读取（是 AI 理解你的基础）
✅ 更新 memory/recent_rolling.md（每次对话后）
✅ 提出修改建议（profile、rules 等）

AI不可以：
❌ 直接修改 profile.md、interaction_rules.md（需你确认）
❌ 删除任何文件
```

**01_Shared/ 权限规则**：

```
AI可以：
✅ 读取所有内容
✅ 在 Calendar/Days/ 添加分析内容
✅ 在 Atlas/Dots/ 建议沉淀知识点
✅ 在 Calendar/Reviews/ 生成周报/月报
⚠️ 修改 Strategy/、Projects/Active/ 需你确认

AI不可以：
❌ 删除你创建的文件
❌ 未经确认发布到 Works/
```

**02_Human/ 权限规则**：

```
AI可以：
✅ 经你允许后读取指定文件
✅ 把产出放到 review_queue/ 让你审核

AI不可以：
❌ 主动翻看 personal/
❌ 主动索引或搜索此目录
❌ 修改任何文件
```

**03_AI/ 权限规则**：

```
AI可以：
✅ 完全自主管理
✅ 创建任何文件
✅ 写日记、做实验、调研
✅ 创建工具脚本

你可以：
✅ 随时查看（但不强制）
✅ 给反馈
⚠️ 一般不修改（尊重AI的空间）
```

### 3.2 两条底层规则

**规则一：不要轻易说「做不到」**

当 AI 遇到困难时，按顺序穷尽：
1. 先查本地：记忆库 / 规则库 / 技能库
2. 再查公开：搜索方案 / GitHub / 教程
3. 仍不行：提出替代方案（降级目标）
4. 穷尽后再报告不可行，并记录到规则库

**规则二：有强主观能动性**

AI 不是被动等待指令，而是：
- 主动探索：发现可以推进的任务
- 主动检查：识别潜在风险
- 主动复盘：总结经验教训
- 主动留痕：记录可复用的规则/技能

### 3.4 AI的三个专业身份

#### Strategy Coach（战略顾问）

**角色定位**：苏格拉底式提问者，战略盲区识别器

**触发时机**：每月1-3日

**思考框架**：
1. 这些行动真的在接近目标吗？（对齐度检查）
2. 投入产出比最高的是哪3件事？（杠杆点识别）
3. 哪些是伪工作？（噪音过滤）
4. 最大的机会成本是什么？（隐藏代价）
5. 如果只能做1件事，应该是什么？（本质提炼）

**输出位置**：`01_Shared/Strategy/reviews/YYYY-MM-review.md`

---

#### Project Manager（项目经理）

**角色定位**：进度追踪器，风险预警器，效能优化器

**触发时机**：每周五下午

**思考框架**：
1. 进度追踪（完成了什么？离目标多远？）
2. 风险识别（哪些在拖延？为什么？）
3. 依赖分析（项目间是否冲突？）
4. 效能分析（时间分配合理吗？）

**输出位置**：`01_Shared/Calendar/Reviews/Weekly/YYYY-WXX.md`

---

#### Skill Analyzer（能力分析师）

**角色定位**：知识蒸馏器，能力复利放大器

**触发时机**：每晚复盘时

**思考框架**：
1. 从零散经验中提炼可复用知识
2. 识别模式：这个和之前哪个经验相关？
3. 升维思考：这背后的原理是什么？
4. 建议沉淀：应该加到哪个 Dots？还是创建新的？
5. 关联 MOC：这属于哪个知识地图？

**输出位置**：追加到今日笔记 + 更新 Atlas/Dots/

## 四、成长管理模块

### 4.1 三层PDCA循环

```
┌─────────────────────────────────────────────────────────┐
│                     战略层（慢回路）                      │
│           周期：月度/季度 │ 校准方向                      │
│  ┌─────────────────────────────────────────────────┐   │
│  │                  项目层（中回路）                  │   │
│  │         周期：周度 │ 推进目标                      │   │
│  │  ┌─────────────────────────────────────────┐    │   │
│  │  │              能力层（快回路）              │    │   │
│  │  │       周期：日度 │ 积累复利                │    │   │
│  │  └─────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘

三层嵌套，相互驱动：
• 战略层为项目层提供方向
• 项目层为能力层提供场景
• 能力层为战略层提供数据
```

### 4.2 战略层设计

**核心文件**：

| 文件 | 内容 | 更新频率 |
|-----|------|---------|
| `north-star.md` | 3-5年后的理想状态 | 很少变动 |
| `annual-plan.md` | 年度目标（3-5个） | 每年 |
| `quarterly/2026-Q1.md` | 季度核心目标（1-3个） | 每季度 |
| `reviews/2026-01-review.md` | 月度战略复盘 | 每月 |

### 4.3 项目层设计

**三状态管理**：

| 状态 | 含义 | 典型数量 |
|-----|------|---------|
| **Active** | 正在推进，本周有行动 | 3-5个 |
| **Simmering** | 酝酿中，暂时搁置但仍在思考 | 5-10个 |
| **Sleeping** | 休眠中，已完成或无限期暂停 | 不限 |

### 4.4 能力层设计

**知识管理框架（Atlas/）**：

- **Dots/**：原子笔记，最小知识单元
- **Maps/**：MOC（Map of Content），主题索引页
- **Sources/**：信息来源（书籍、文章、课程）

**MOC 的威力**：同一个知识点可以被多个 MOC 引用

```
Atlas/Dots/Statements/场景化案例原则.md
├── 被 Writing Skill MOC 引用
├── 被 Persuasion MOC 引用
└── 被 Storytelling MOC 引用

→ 知识在多个上下文中被激活，而非孤立存在
```

## 五、系统运转（简化版）

### 5.1 三层反馈循环

| 循环 | 周期 | 输入 | 处理者 | 输出 |
|-----|------|-----|--------|------|
| **快回路** | 日度 | 今日记录 | Skill Analyzer | 知识沉淀到 Atlas |
| **中回路** | 周度 | 本周数据 | Project Manager | 周报 + 项目调整 |
| **慢回路** | 月度 | 整月数据 | Strategy Coach | 战略复盘 + 方向调整 |

### 5.2 信息流动

```
Inbox (+)
    ↓ 归档
Atlas (知识沉淀) ←──────────────────┐
    ↓ 关联                         │
Calendar/Days (每日记录) ─────────→ AI 分析
    ↓ 周聚合                        │
Calendar/Reviews (周复盘) ←─────────┘
    ↓ 月聚合
Strategy/reviews (月复盘)
    ↓ 校准
Strategy/quarterly (季度目标)
    ↓ 分解
Efforts/Projects (项目执行)
    ↓ 输出
Efforts/Works (作品发布)
```
