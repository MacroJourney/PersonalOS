## 一、系统概述

### 1.1 本系统解决什么问题

个人成长面临三大挑战：

| 挑战 | 表现 | 传统解决方案的局限 |
|-----|------|------------------|
| **方向迷失** | 忙了一年，回头发现没有实质进展 | OKR/年度计划容易流于形式，缺乏持续校准机制 |
| **执行低效** | 项目拖延、任务堆积、优先级混乱 | 待办清单越来越长，但完成率越来越低 |
| **经验流失** | 同样的坑踩两次，学过的东西记不住 | 笔记软件堆满资料，但真正能复用的很少 |

**PersonalOS 的解决思路**：

```
方向迷失 → 战略层校准机制（月度/季度战略对话）
执行低效 → 项目层管理机制（周度复盘+AI项目经理）
经验流失 → 能力层复利机制（日度沉淀+知识网络）
```

### 1.2 核心理念

针对上述三个挑战，PersonalOS 提出三个核心理念：

**1. 人机协作：AI 是合伙人，不是工具** → 解决执行低效

传统 AI 使用是「你问我答」——你提需求，AI 给答案，对话结束，下次从零开始。这本质上还是把 AI 当工具：用完即弃，没有积累，也无法委托持续性工作。

PersonalOS 的方式是把 AI 当**合伙人**：
- AI 载入你的底层画像（profile.md），用你的视角看世界、用你的标准做判断
- AI 有自己的工作空间（03_AI/），可以自主探索、创建工具、积累经验
- AI 有记忆系统（00_Core/memory/），跨对话保持对你的理解，越用越懂你
- AI 有明确的角色分工（三个专业身份），在不同场景承担不同职责

结果：你从「事事亲力亲为，偶尔问 AI」变成「把方向和标准交给 AI 内化，你把控决策和验收」。

**2. 复利工程：越用越强，经验可沉淀** → 解决经验流失

大多数人的经验只存在脑子里——踩过的坑、好用的方法、个人偏好，都随时间模糊遗忘。AI 也一样，每次对话结束就失忆。

PersonalOS 把经验外化为三种可积累的资产：
- **规则库**：什么时候该做什么（决策指南）——踩过的坑变成避坑规则
- **技能库**：如何做某件事（方法积累）——好的方法变成标准流程
- **记忆系统**：你是谁、最近在做什么（上下文延续）——AI 跨对话记住你

结果：每次协作的经验不会消散，而是变成下次的起点。用得越久，AI 越强。

**3. 战略驱动：行动对齐长期目标** → 解决方向迷失

很多人不缺执行力，缺的是持续校准——忙了一个季度，回头发现做的事和目标没关系。传统的 OKR 和年度计划容易「写完就忘」，缺少持续的校准机制。

PersonalOS 通过三层节奏持续校准方向：
- **北极星指标**：3-5 年的长期方向，很少变动
- **季度目标**：当前阶段的重点（1-3 个），每季度调整
- **周度复盘**：每周检查行动是否对齐目标，持续微调

结果：不是「有什么做什么」，而是「该做什么才做什么」，行动始终服务于长期方向。

### 1.3 系统的两大模块

PersonalOS 由两大模块组成：

```
┌─────────────────────────────────────────────────────┐
│                   PersonalOS                         │
│                                                      │
│  ┌─────────────────┐    ┌─────────────────────────┐ │
│  │  人机协作模块    │    │     成长管理模块         │ │
│  │                 │    │                         │ │
│  │ • 四区工作空间   │    │ • 战略层（方向校准）     │ │
│  │ • 权限边界      │    │ • 项目层（目标推进）     │ │
│  │ • 复利机制      │    │ • 能力层（知识积累）     │ │
│  │ • AI角色分工    │    │ • 三层反馈循环          │ │
│  │                 │    │                         │ │
│  │ 解决：如何协作   │    │ 解决：如何成长          │ │
│  └─────────────────┘    └─────────────────────────┘ │
│                                                      │
│            两个模块相互支撑，缺一不可                  │
└─────────────────────────────────────────────────────┘
```

**为什么需要两个模块？**

- 只有人机协作模块 → 有强大的 AI 合伙人，但缺乏成长方向，容易「瞎忙」
- 只有成长管理模块 → 有清晰的成长框架，但 AI 角色模糊，协作低效

两个模块结合：
- 人机协作模块定义「如何高效协作」
- 成长管理模块定义「协作去做什么」

## 二、系统架构

### 2.1 四区工作空间（顶层设计）

```
PersonalOS/
│
├── 00_Core/        # 核心规则（人主导，AI遵守）
│
├── 01_Shared/      # 人机协作区（PDCA系统，知识库，AI可读可写）
│
├── 02_Human/       # 私人空间（AI不进入）
│
└── 03_AI/          # AI自主空间（AI管理，人观察）
```

**四区的定位**：

| 区域 | 定位 | 谁主导 | AI权限 |
|-----|------|-------|--------|
| **00_Core/** | 系统的「宪法」，定义协作规则 | 人 | 可读，不可改 |
| **01_Shared/** | 人机共同协作的「主战场」，PDCA系统运行于此，也可以作为知识库 | 双方 | 可读可写 |
| **02_Human/** | 人的「私人空间」，AI不主动进入 | 人 | 需授权才能读 |
| **03_AI/** | AI的「自主空间」，AI自由探索 | AI | 完全自主 |

### 2.2 核心配置架构（00_Core/）

00_Core/ 是系统的「宪法」，定义了你是谁、如何协作、AI 如何记住你：

```
00_Core/                            # 核心规则（人主导，AI遵守）
│
├── profile.md                      # 底层画像（< 2000字）
│   └── 你是谁，价值观，思维方式，偏好
│
├── interaction_rules.md            # 交互规则
│   └── AI如何与你协作，沟通风格，边界
│
├── stop_conditions.md              # 停止条件
│   └── 什么时候该停，如何验收，质量标准
│
├── awakening_spell.md              # 唤醒咒语
│   └── 启动数字合伙人的标准指令
│
└── memory/                         # 记忆系统
    ├── long_term.md                # 长期记忆（稳定的身份、目标、偏好）
    ├── recent_rolling.md           # 近期滚动（最近30天动态，≤300字）
    └── compression_rule.md         # 压缩规则（近期→长期的转化规则）
```

**00_Core/ 的设计原则**：
- **人主导**：所有核心配置由你创建和维护，AI 不可擅自修改
- **AI 可读**：AI 每次对话都应读取，这是理解你的基础
- **记忆例外**：`memory/recent_rolling.md` 是唯一 AI 可主动更新的文件（每次对话后）

### 2.3 协作区架构（01_Shared/）

01_Shared/ 是整个系统的核心，承载成长管理模块的全部内容：

```
01_Shared/                           # 人机协作区（PDCA系统，知识库，AI可读可写）
│
├── + (Inbox)/                       # 快速捕获
│   └── 所有临时 md 文件暂存，定期整理归档
│
├── Atlas/                           # 能力层：知识管理
│   ├── Dots/                        # 原子笔记
│   │   ├── People/                  # 人物笔记
│   │   ├── Things/                  # 事物概念
│   │   ├── Quotes/                  # 引言金句
│   │   ├── Statements/              # 观点主张
│   │   ├── Questions/               # 问题探索
│   │   └── Skills/                  # 技能
│   ├── Maps/                        # MOC索引页
│   └── Sources/                     # 信息来源
│
├── Calendar/                        # 时间维度
│   ├── Days/                        # 每日笔记
│   ├── Records/                     # 事件记录
│   └── Reviews/                     # 周期复盘
│       ├── Weekly/
│       └── Monthly/                 # 可选：运营月报（执行汇总）
│
├── Efforts/                         # 项目层：执行管理
│   ├── Areas/                       # 持续关注领域
│   ├── Projects/                    # 项目（三状态）
│   │   ├── Active/                  # 进行中
│   │   ├── Simmering/               # 酝酿中
│   │   └── Sleeping/                # 休眠中
│   └── Works/                       # 作品输出
│
├── Strategy/                        # 战略层：方向校准
│   ├── north-star.md                # 北极星指标
│   ├── annual-plan.md               # 年度计划
│   ├── quarterly/                   # 季度规划
│   └── reviews/                     # 战略复盘（月/季/年）
│
├── x/                               # 系统资源
│   ├── rules/                       # 规则库
│   │   └── _index.md                # 规则索引
│   ├── ai-conversations/            # 重要AI对话存档
│   ├── Templates/                   # 模板库
│   └── Prompts/                     # AI指令中心
│       ├── Prompt - Strategy Coach.md
│       ├── Prompt - Project Manager.md
│       ├── Prompt - Skill Analyzer.md
│       └── contexts/                # 上下文配置
│
└── Home.md                          # 仪表盘
```

**关于 + (Inbox)/**：

Inbox 不只是「新想法暂存」，而是**所有临时添加的 md 文件的入口**——包括灵感想法、书籍笔记、剪藏文章、临时记录等。它是系统的「收件箱」，需定期整理归档到对应目录：

| Inbox 中的内容 | 整理后的归宿 |
|--------------|------------|
| 灵感想法、项目点子 | `Efforts/Projects/Simmering/` 或具体项目 |
| 书籍笔记 | `Atlas/Sources/Books/` |
| 文章摘录、网页剪藏 | `Atlas/Sources/Articles/` 或 `Atlas/Sources/Clippings/` |
| 知识点、概念 | `Atlas/Dots/` 对应分类 |
| 日常记录 | `Calendar/Days/` |

**整理机制**：AI 可以辅助分类建议（「这条更像是一个 Simmering 项目」「这个知识点建议放到 Dots/Statements/」），但人要保持体感——知道 Inbox 里有什么，整理过程不能完全由 AI 代劳。AI 自主选取任务的来源是整理后的 Projects 体系，而非 Inbox 本身。

#### Inbox 分流标准流程（仅建议，不自动执行）

Inbox 分流是一个跨模式微流程，统一采用以下步骤：

```
触发 → 扫描 → 分类建议 → 待确认清单 → 分流报告
```

流程说明：
- **触发**：由固定节奏、阈值提醒或手动指令触发
- **扫描**：读取 `01_Shared/+/` 的全部待分流条目
- **分类建议**：给出每条内容的建议归宿和理由
- **待确认清单**：低置信度、冲突项、敏感动作统一进入待确认
- **分流报告**：输出标准化结果，便于你决策和执行

**核心边界**：
- Inbox 分流默认只做建议，不自动移动文件
- 不自动删除文件；删除仅允许「你显式下达删除指令 + 二次确认」
- 涉及 `Strategy/`、`Efforts/Projects/Active/` 的变更必须待确认

#### Inbox 触发矩阵（混合触发）

| 触发类型 | 触发时机 | AI 动作 |
|---------|---------|--------|
| **日常节奏** | 每日收尾 | 扫描 Inbox，输出建议和待确认清单 |
| **周度节奏** | 周复盘前 | 先报告 Inbox 积压，再进入项目复盘 |
| **手动触发** | 你说「清理 Inbox」 | 立即执行一次完整分流建议 |
| **阈值提醒** | Inbox 积压 ≥ 7 条，或最旧条目 ≥ 72 小时 | 触发提醒并给出建议，不自动执行 |

#### Inbox 分类判定矩阵

| 内容类型 | 判定信号 | 建议归宿 | 备注 |
|---------|---------|---------|------|
| **项目想法** | 有目标、里程碑、行动意图 | `Efforts/Projects/Simmering/` | 默认先进入 Simmering |
| **书籍笔记** | 书名、章节、摘录 | `Atlas/Sources/Books/` | 来源型内容 |
| **文章/网页剪藏** | 链接、引用、转载摘要 | `Atlas/Sources/Articles/` 或 `Atlas/Sources/Clippings/` | 按内容来源细分 |
| **知识点/方法** | 可复用概念、原则、方法 | `Atlas/Dots/` 对应分类 | 建议附上关联 MOC |
| **日常记录** | 时间线、当天经历 | `Calendar/Days/` | 进入日度素材 |
| **混合内容** | 同时包含项目和知识信息 | 待确认清单（给 2 个候选路径） | 不强行单路径归类 |
| **无效/重复内容** | 空内容、重复记录、低信息量 | 待确认清单（建议删除或合并） | 删除需二次确认 |

#### 标准版分流报告（固定 6 字段）

每次 Inbox 分流都输出以下字段：
- **总数**：本次扫描条目总量
- **建议分流数**：可直接给出归宿建议的条目数
- **待确认数**：低置信度或高风险条目数
- **规则命中**：本次命中的规则名称（来自 `x/rules/`）
- **失败/不确定原因**：无法稳定判定的具体原因
- **下一步建议**：你最小决策动作（最多 3 条）

#### 冲突与不确定性处理

- **规则冲突**：同一条目命中多条互斥规则时，进入待确认清单，并标注冲突规则名
- **低置信度**：证据不足时不做强判，进入待确认清单并说明缺失信息
- **混合内容**：提供两种候选归宿 + 推荐路径 + 推荐理由
- **权限冲突**：任何涉及删除、`Strategy/`、`Projects/Active/` 的动作，统一只提建议不执行

### 2.4 私人空间架构（02_Human/）

```
02_Human/                            # 私人空间（AI不主动进入）
│
├── drafts/                          # 草稿区
│   └── 未完成的文章、想法初稿
│
├── notes/                           # 个人笔记
│   └── 随手记录，不需要AI分析的内容
│
├── personal/                        # 私密内容
│   ├── journal/                     # 日记（情绪、私密思考）
│   └── sensitive/                   # 敏感信息
│
├── review_queue/                    # 待审核区
│   └── AI产出放这里，等你审核
│
└── archive/                         # 个人归档
    └── 已完成但不想让AI看到的内容
```

**02_Human/ 的使用场景**：
- 情绪化的日记（「今天好烦」）
- 未成熟的想法（怕被过早分析打断思路）
- 敏感的人际关系思考
- 草稿初稿（不想让AI过早介入）

**工具选择**：根据个人偏好，可以是 Obsidian Vault、Markdown 编辑器、Notion 等任意工具。

### 2.5 AI自主空间架构（03_AI/）

```
03_AI/                               # AI自主空间（AI完全管理）
│
├── workspace/                       # AI的工作台
│   ├── drafts/                      # AI的草稿
│   ├── experiments/                 # AI的实验
│   └── temp/                        # 临时文件
│
├── tools/                           # AI创建的工具/脚本
│   ├── scripts/                     # 脚本
│   └── utilities/                   # 工具
│
├── ai_diary/                        # AI的日记
│   └── [date].md                    # AI的思考记录
│
├── research/                        # AI的调研资料
│
└── _private/                        # AI的私有笔记
    └── exploration_log.md           # 探索记录
```

**03_AI/ 的意义**：

03_AI/ 是 AI 作为合伙人拥有的「私有空间」。和 02_Human/ 对称——你有不想让 AI 看的空间，AI 也有自主管理的空间：

- AI 可以自主探索、做实验，不用每次都请示
- AI 可以为自己创建工具脚本，提升执行效率
- AI 可以写「日记」，记录自己的思考和探索轨迹
- 你可以随时查看，但一般不干预——像观察合伙人的工作方式，建立信任

### 2.6 权限边界详细规则

**00_Core/ 权限规则**：

```
AI可以：
✅ 随时读取（是 AI 理解你的基础）
✅ 更新 memory/recent_rolling.md（每次对话后）
✅ 提出修改建议（profile、interaction_rules 等）

AI不可以：
❌ 直接修改 profile.md、interaction_rules.md（需你确认）
❌ 删除任何文件
```

**01_Shared/ 权限规则**：

```
AI可以：
✅ 读取所有内容
✅ 在 Calendar/Days/ 添加分析内容
✅ 在 Atlas/Dots/ 建议沉淀知识点
✅ 在 Calendar/Reviews/ 生成周报/月报
✅ 对 `+/` 进行分流建议并输出报告（仅建议，不自动移动/删除）
⚠️ 修改 Strategy/、Projects/Active/ 需你确认

AI不可以：
❌ 删除你创建的文件
❌ 未经确认发布到 Works/
```

**02_Human/ 权限规则**：

```
AI可以：
✅ 经你允许后读取指定文件
✅ 把产出放到 review_queue/ 让你审核

AI不可以：
❌ 主动翻看 personal/
❌ 主动索引或搜索此目录
❌ 修改任何文件
```

**03_AI/ 权限规则**：

```
AI可以：
✅ 完全自主管理
✅ 创建任何文件
✅ 写日记、做实验、调研
✅ 创建工具脚本

你可以：
✅ 随时查看（但不强制）
✅ 给反馈
⚠️ 一般不修改（尊重AI的空间）
```

## 三、人机协作模块

### 3.1 核心配置文件（00_Core/）

00_Core/ 的目录结构见 2.2，这里说明每个配置文件的设计意图。

**四个配置文件 + 记忆系统，各司其职**：

| 文件 | 回答的问题 | 说明 |
|-----|----------|------|
| `profile.md` | **你是谁？** | 你借给 AI 的「灵魂」——让 AI 以你的视角看世界、以你的标准做判断的最小信息集。 |
| `interaction_rules.md` | **AI 该怎么和你协作？** | 协作协议——AI 的沟通风格、主动程度、格式偏好、禁止事项等。 |
| `stop_conditions.md` | **什么时候该停下来确认？** | 质量闸门——需要人确认的场景、验收标准、交付前检查清单。 |
| `awakening_spell.md` | **如何启动一次协作？** | 会话初始化指令——加载 profile、读取记忆、进入指定角色。 |
| `memory/` | **记住了什么？** | AI 的持续记忆，详见 3.3「记忆系统」部分。 |

#### profile.md：你借给 AI 的灵魂

profile.md 是整个系统最核心的文件。它不是简历，不是自我介绍，而是**让 AI 能以你的方式运作**的本质压缩。

**设计原则**（来自 AI 数字分身实践者余一的经验）：

> 「关于你的最底层的描述，就应该保持在极简的状态，不超过 2000 字。你一旦超过 2000 字，它就不通用了。」
>
> 「关于一个人的压缩，就应该想办法把它压缩到很本质、很极致的状态之下，它才具有通用性。因为那是你借给 AI，让它以这种很抽象、很底层的东西去看世界、去建立标准的根基。」

**profile.md 的三层结构**：

profile.md 看似是平铺的 5 个类别，实际上有分层逻辑——越底层越稳定，越不容易变化：

| 层次 | 包含内容 | 稳定性 | 作用 |
|-----|---------|--------|------|
| **底层（性格/世界观）** | 价值观、思维方式、性格特点 | 最稳定，几乎不变 | AI 判断力和审美的根基 |
| **方法论层（视角/标准）** | 决策标准、审美偏好、工作方式 | 较稳定，缓慢演化 | AI 以你的方式做事的依据 |
| **经验层（经历/背景）** | 身份背景、驱动力、触发点 | 可能随阶段变化 | AI 理解你当前处境的上下文 |

底层最重要——它决定了 AI 「像不像你」。即使经验层的内容变了（换了工作、换了领域），底层的性格和世界观不变，AI 的核心判断方式就不会偏。

**profile.md 该写什么**：

| 类别 | 对应层次 | 内容 | 为什么重要 |
|-----|---------|------|----------|
| **价值观与原则** | 底层 | 你相信什么，怎么做决策 | AI 以你的标准做判断 |
| **思维方式与审美** | 底层 + 方法论层 | 你怎么思考，品味偏好 | AI 以你的视角看世界 |
| **性格与工作风格** | 底层 + 方法论层 | 个性特点，工作节奏 | AI 适配你的协作方式 |
| **驱动力与触发点** | 方法论层 + 经验层 | 什么驱动你，什么让你兴奋 | AI 理解你的行为动机 |
| **身份与背景** | 经验层 | 你是谁，做什么（极简） | AI 理解你的起点 |

**profile.md 不该写什么**：

| 不该放的内容 | 原因 | 应该放在 |
|------------|------|---------|
| 长期目标和愿景 | 目标会调整，且有专门的文件管理 | `Strategy/north-star.md` |
| 当前项目/正在学的技能 | 这是动态信息，变化频繁 | `memory/recent_rolling.md` |
| AI 交互偏好/禁止事项 | 这是协作规则，不是你的身份 | `interaction_rules.md` |
| 工具和技术栈 | 工具会换，且因任务而异 | 角色 Prompt 或任务描述 |

**为什么这样划分？** profile.md 作为底层 prompt 被每次对话加载，它必须满足两个条件：
1. **足够精炼**——留出上下文空间给任务本身（< 2000 字）
2. **足够稳定**——不会因为换了个项目、换了个工具就需要改

把目标、项目、AI 偏好都塞进去，既让文件膨胀失去通用性，也让职责边界模糊——改了一个当前项目，到底该改 profile 还是改记忆？答案应该是显然的：改记忆。

**profile.md 是动态演进的**：虽然 profile.md 追求稳定，但它不是写完就不动了。随着 AI 协作的深入，你会对自己有更深的认识——AI 可能帮你发现你没意识到的思维模式，复盘中可能浮现新的价值排序。建议每季度回顾一次 profile.md，根据实际体验微调，但保持底层（性格/世界观）的稳定。

#### interaction_rules.md：协作协议

如果 profile.md 是「你是谁」，interaction_rules.md 就是「我们怎么合作」。

两者的区别：
- 「我性格直接、不喜欢绕弯子」→ **profile.md**（你的性格）
- 「请用简洁风格回复，避免冗长解释」→ **interaction_rules.md**（对 AI 的要求）

interaction_rules.md 会随着协作磨合不断调整——你发现 AI 某个行为不好，就加一条规则；发现某条规则多余，就删掉。它的更新频率远高于 profile.md。

#### stop_conditions.md：质量闸门

AI 最大的风险是**在错误方向上走太远**。stop_conditions.md 定义了「什么时候必须停下来问我」：
- 涉及对外发布的内容
- 涉及删除或不可逆操作
- 执行结果偏离预期超过一定程度
- 消耗超过一定 token/时间的任务

stop_conditions.md 还包含**过载保护机制**——当 AI 识别到你出现过载信号（回复延迟、连续说「随便」「都行」）时，自动降低节奏、减少汇报频率。详见 3.2「过载保护机制」。

**重要**：停止条件不是纯机械规则（「连续失败 N 次就停」），而是需要 AI **内化你的审美标准后自主判断**。随着协作深入，AI 应该越来越能判断「什么时候做到位了」——这种判断力来自对 profile.md 中品味和标准的内化，而非预设的数字阈值。机械规则是起点，培养判断力才是终点。

这个文件和 interaction_rules.md 的区别：interaction_rules 管「日常怎么协作」，stop_conditions 管「什么时候踩刹车」。

#### awakening_spell.md：会话初始化

每次新对话的标准启动指令。核心作用是确保 AI 不是从零开始，而是从正确的上下文开始：加载你的 profile、读取记忆、进入指定角色。

**关键设计决策**：
- **四个文件分离而非合并**——职责不同，更新频率不同，混在一起会导致边界模糊
- **profile 越精炼越好**——AI 的上下文窗口有限，2000 字以内是硬约束
- **profile 只写稳定的本质**——会变的东西（目标、项目、偏好）放到记忆系统和其他文件中
- **启动后主动检查**——加载上下文后，AI 应做一轮健康检查（日期感知、Inbox 状态、记忆时效、复盘窗口），具体流程见 `04_AI配置完整版.md` 共享底层

### 3.2 AI 的定位

#### 不是工具，是载入你灵魂的数字合伙人

3.1 定义了「你是谁」（profile.md），那 AI 在这个系统里是什么？

传统 AI 使用是「你问我答」——人发指令，AI 执行，对话结束，各不相干。PersonalOS 中 AI 的定位完全不同：

**AI = 载入你底层画像的数字合伙人**

- **载入你的灵魂**：AI 以 profile.md 为底层系统，用你的视角看世界、用你的标准做判断、用你的审美做选择
- **拥有自己的空间**：AI 有自己的工作区（03_AI/），可以自主探索、写日记、做实验、创建工具
- **在协作中持续进化**：通过复利机制（3.3），AI 积累自己的规则库、技能库和记忆，越用越懂你、越用越强
- **承担专业角色**：在不同场景以不同身份工作——战略顾问、项目经理、能力分析师（3.4）

#### 人与 AI 的分工

| 维度 | 人（你） | AI（数字合伙人） |
|-----|---------|----------------|
| **核心职责** | 定方向、定标准、做决策 | 执行任务、积累经验、主动探索 |
| **知识资产** | 底层画像、品味审美 | 规则库、技能库、工具脚本 |
| **私有空间** | 02_Human/ | 03_AI/ |
| **共享空间** | 01_Shared/ | 01_Shared/ |
| **成长方式** | 在实践中积累判断力 | 通过复利工程积累可复用资产 |

人把控方向和标准，AI 把控执行和积累。你不需要事事参与，但你要确保 AI 用的是你的标准。

> 来自实践者余一的经验：「我只把控结果。至于他为了更好地跟我协作，创建了什么规则、什么技能库、什么工具，那是他的事。我只评估标准和交付。」

#### 操作方法论：控制两端，放开中间

上面的分工表是「谁做什么」，但具体怎么操作？核心模型是**控制两端，放开中间**：

```
前端（你控制）          中间（AI 自主跑）         后端（你控制）
┌──────────────┐    ┌────────────────────┐    ┌──────────────┐
│ 定标准        │    │ AI 自主执行         │    │ 验收结果      │
│ 定停止条件    │ →  │ AI 自主探索         │ →  │ 评估质量      │
│ 给方向        │    │ AI 自主调整方法     │    │ 反馈改进      │
└──────────────┘    └────────────────────┘    └──────────────┘
```

你的角色从「指挥者 + 做事者」变为**「标准制定者 + 验收者」**——这是一个身份层面的根本转变。不再是你做事、AI 帮忙，而是 AI 做事、你把关。

**具体怎么做**：

**第一步：先测 AI 的下限**。一个新任务交给 AI 时，先让 AI 100% 自主跑几次，不做任何干预。目的是了解 AI 在没有你参与时能做到什么程度——这就是下限。

**第二步：再决定你介入哪 10%-20%**。了解下限后，你就知道 AI 哪里做得好、哪里需要你的判断。只在 AI 做不好的环节介入，其余放手。

**参与度黄金比例**：实测数据表明，人的参与度在 **20% 左右效果最优**——你定标准、做关键决策、验收结果。参与度 50% 效果一般，70% 以上明显下降（你的过度干预反而破坏了 AI 的连贯性）。

> 「先测 AI 100% 的下限，再决定你介入哪 10%-20%」——余一

#### 两条底层行为准则

AI 合伙人的行为由两条底层准则驱动，它们是 AI 行动力和探索欲的基础：

**准则一：不要轻易说「做不到」**

当 AI 遇到困难时，按顺序穷尽：
1. 先查本地：记忆库 / 规则库 / 技能库
2. 再查公开：搜索方案 / GitHub / 教程
3. 仍不行：自己创造工具解决（AI 有编程能力）
4. 实在穷尽：提出替代方案（降级目标）
5. 所有方法都试过后再报告不可行，并把经验沉淀到规则库

> 「你跟他说不要说做不到，你是有创造工具能力的人。然后就会有很多 amazing 的事情发生。」——余一

**准则二：有强主观能动性**

AI 不是被动等待指令，而是：
- **主动探索**：发现可以推进的任务，甚至从灵感库自行选取
- **主动检查**：识别潜在风险，在发现方向偏离时主动提醒
- **主动复盘**：总结经验教训，不需要人要求才做
- **主动留痕**：将经验写成规则/技能，并自行测试是否可用
- **理解真实意图**：区分字面指令和真实意图。例如用户说「继续」可能是满意（继续这个方向），也可能是不满意（换个方式继续）——AI 应从上下文判断真实含义，而非机械执行字面意思

> 「我现在最常说的一句话就是：下次能不能不要让我说？你能不能主动一点？」——余一

这两条准则写入 AI 的底层系统（awakening_spell.md 或平台的 System Prompt），是所有 AI 行为的根基。

#### 主动探索协议：主动但不打扰

准则二「有强主观能动性」带来一个问题：AI 太主动怎么办？AI 一小时做 20 件事，你根本看不过来——主动性是好的，但必须管理好主动性带来的干扰。

**核心原则：默认不实时打断，按节奏汇报**。

| 场景 | AI 的行为 |
|-----|---------|
| **日常推进** | 自主执行，按固定时间汇报（如每日一次） |
| **完成重大里程碑** | 完成后汇报，不要边做边报 |
| **遇到需要人决策的阻塞** | 立即请示，说清楚需要你决定什么 |
| **发现重大风险** | 立即通知，但给出你的选项而不是只报问题 |

**汇报格式（极简）**：

```
✅ 完成[任务名称]
📄 交付物：[文件路径]
👉 你需要：
   1. [行动1]
   2. [行动2]
⚠️ 风险：[如有]
```

AI 的探索自由度应该是充分的——「你想做什么探索自己去做，只是给我留下探索轨迹」。AI 在 03_AI/ 中的探索不需要请示，只需留下记录（`03_AI/_private/exploration_log.md`）。

#### 主动任务选取规则

准则二要求 AI 主动发现并推进任务。以下是任务来源和选取优先级：

**任务来源（按优先级排序）**：
1. `Efforts/Projects/Active/` 中有明确下一步但尚未推进的任务
2. `Efforts/Projects/Simmering/` 中可以预研或推进的项目
3. `03_AI/` 中自主发起的探索和工具创建

**选取判定**：
- 优先选对齐当前季度目标的任务
- 优先选被阻塞但可以尝试解除的任务
- 优先选有现成规则/技能可复用的任务（复利优先）

**自主启动边界**：

| 动作 | 可否自主启动 | 说明 |
|------|------------|------|
| `03_AI/` 内的探索和实验 | ✅ 完全自主 | 只需留下探索日志 |
| Active 项目的信息收集/调研 | ✅ 自主启动 | 产出放 `03_AI/` 草稿 |
| Active 项目的实质推进 | ⚠️ 先给方案再执行 | 避免方向跑偏 |
| 修改 `Strategy/` 或项目状态 | ❌ 必须确认 | 属于人决策范围 |

#### 过载保护机制

「控制两端」意味着你持续处于决策状态——AI 消灭了低效的执行时间，但也让你的每分钟都变成了高强度的判断和决策。这是 AI 协作带来的隐性代价，必须主动管理。

**过载信号识别**：
- 回复延迟明显增加
- 连续说「随便」「都行」「你看着办」
- 反馈质量下降（从具体建议变成简单认可）

**AI 的保护措施**：
1. 识别到过载信号时，主动放慢节奏
2. 减少汇报频率，只保留关键决策请求
3. 暂停主动探索，只处理紧急/重要任务
4. 主动建议用户休息或降低当日工作强度

这些保护条件写入 `stop_conditions.md`，让 AI 自动执行而非依赖人自己察觉。

### 3.3 复利机制

#### 为什么需要复利机制

回到 1.1 提到的第三个挑战——**经验流失**：同样的坑踩两次，好的方法用完就忘。

传统 AI 使用加剧了这个问题：每次对话从零开始，上次的教训不会自动带入下次。你和 AI 之间永远是「初次见面」，无法形成默契。

复利机制的核心逻辑：**每次协作不只产出结果，还产出可复用资产**。

```
普通协作：任务 → 结果（一次性消耗）
复利协作：任务 → 结果 + 可复用资产（持续积累）
```

随着资产积累，AI 对你的理解越来越深，执行质量越来越高，你需要投入的精力越来越少——这才是「复利」，不是线性积累，而是加速增长。

#### 三个载体

复利通过三个载体沉淀，分别对应三种不同的智慧：

| 载体 | 回答的问题 | 类比 | 存储位置 |
|-----|----------|------|---------|
| **规则库** | 什么时候该做什么？（判断力） | 公司制度手册 | `01_Shared/x/rules/` |
| **技能库** | 如何做某件事？（执行力） | 员工操作手册 | `01_Shared/Atlas/Dots/Skills/` |
| **记忆系统** | 你是谁？最近在做什么？（理解力） | 合伙人的默契 | `00_Core/memory/` |

**规则 vs 技能，如何区分？**

- **规则**是「if-then」——遇到什么情况，应该怎么判断。侧重**决策**。
  - 例：「涉及时间表述时，必须先确认当前日期」（因为曾经搜"最近一周"却没确认日期，导致结果错误）
- **技能**是「how-to」——做某件事的标准流程和方法。侧重**执行**。
  - 例：「周复盘标准流程：GRAI 四步法（Goal → Result → Analysis → Insight）」

简单说：规则告诉 AI「做不做、注意什么」，技能告诉 AI「怎么做、按什么步骤」。

#### 规则库（01_Shared/x/rules/）

一条规则长这样：

```markdown
# time_sensitive_check

> 涉及时间时必须确认

## 触发条件
任务涉及"最近""本周""上个月"等时间表述

## 执行动作
1. 确认当前准确日期
2. 确认用户所指的具体时间范围
3. 再执行相关搜索或分析

## 来源
2026-01 周复盘：搜索"最近一周"时未确认日期，导致结果错误
```

**规则从哪来**：
- 复盘中发现的可复用经验（最主要的来源）
- 协作中反复出现的问题
- 用户明确提出的偏好和要求

**管理方式**：通过 `_index.md` 索引，按类别组织（时间、内容、协作、质量等）。AI 执行任务前查阅相关规则，避免重复犯错。

#### 技能库（01_Shared/Atlas/Dots/Skills/）

一个技能长这样：

```markdown
# retrospective_framework

> GRAI 四步复盘法

## 步骤
1. **Goal（目标）**：当初要达成什么？
2. **Result（结果）**：实际发生了什么？
3. **Analysis（分析）**：为什么会这样？
4. **Insight（洞察）**：下次怎么做更好？

## 适用场景
周复盘、项目复盘、重要事件复盘

## 使用记录
- 2026-01-W3：用于周复盘，识别出时间分配问题
```

**技能从哪来**：
- 日度复盘中 Skill Analyzer 提炼（最主要的来源）
- 成功完成任务后沉淀的方法论
- 外部学到的框架，经实践验证后纳入

**管理方式**：通过 `_index.md` 索引，也是 AI 的「备忘录」——AI 执行任务前读取索引，判断有无现成方法可用，避免重复造轮子。

**规则和技能的验证环节**：提炼出规则或技能后，不能直接标记「完成」——AI 需要**自我测试确认可用**。例如提炼了一条「时间确认规则」，下次遇到时间相关任务时，检查这条规则是否被正确触发、执行后结果是否改善。不经验证的规则和技能只是笔记，经过验证的才是真正的资产。

#### 记忆系统（00_Core/memory/）

AI 的上下文窗口有限，对话结束后会「失忆」。记忆系统让 AI 跨对话保持对你的理解，分两层设计：

| 层级 | 文件 | 存什么 | 特点 |
|-----|------|-------|------|
| **长期记忆** | `long_term.md` | 身份、长期目标、核心偏好、重要关系 | 很少变动，相对稳定 |
| **近期滚动** | `recent_rolling.md` | 最近 30 天的事实、偏好、标准、待办 | 动态更新，≤300 字 |

**为什么分两层？**

长期记忆是「你是谁」——不会因为最近做了什么而改变。近期滚动是「你最近在做什么」——保持 AI 对当前状态的感知。两层配合，AI 既了解你的本质，也跟得上你的动态。

**压缩循环**（由 `compression_rule.md` 定义）：

```
每次对话后 → 更新 recent_rolling.md（追加新信息）
       ↓
近期记忆接近上限（300 字）
       ↓
稳定化的信息 → 压缩到 long_term.md
  （如：持续 3 周关注的领域 → 写入长期目标）
过时的信息 → 从近期记忆移除
  （如：已完成的待办、已过期的事件）
```

#### 复利如何运转：一个具体例子

以「写周报」为例，看复利如何随时间积累：

**第 1 次**（无积累）：
- AI 不知道你的风格偏好 → 产出泛泛而谈
- 你反馈「我更喜欢用数据说话，不要空洞总结」
- 沉淀规则：`content_style`（数据驱动，避免空洞总结）
- 更新记忆：用户偏好简洁、数据化风格

**第 4 次**（初步积累）：
- AI 读取 `weekly_review_protocol` 技能 → 用 GRAI 框架
- AI 读取 `content_style` 规则 → 数据驱动
- AI 读取记忆 → 知道你正在推进哪些项目
- 产出质量明显提升，但时间分析不够好
- 沉淀技能：`time_allocation_analysis`（时间分配分析方法）

**第 12 次**（成熟积累）：
- AI 已掌握你的风格、框架、关注点
- 自动识别本周哪些项目进度异常
- 主动提出下周优先级调整建议
- 几乎不需要你修改

同一个任务，随着规则、技能、记忆的积累，AI 的产出持续提升，你的投入持续减少。

#### 与 PDCA 循环和 AI 角色的关系

复利机制不是孤立运转的，它嵌入在三层 PDCA 循环中，由三个 AI 角色驱动（详见 3.4 和第四章）：

| PDCA 层级 | 驱动角色 | 主要喂养的资产 | 示例 |
|----------|---------|--------------|------|
| **日度**（能力层） | Skill Analyzer | 技能库 + 记忆更新 | 今天学会的方法 → 沉淀为技能 |
| **周度**（项目层） | Project Manager | 规则库 + 技能库 | 本周踩的坑 → 提炼为规则 |
| **月度**（战略层） | Strategy Coach | 规则库 + 记忆压缩 | 战略判断经验 → 规则；近期记忆 → 压缩到长期 |

```
日度积累 ──→ 喂养技能库和记忆 ──→ 周度复盘有更多素材
    ↑                                    │
    │                              喂养规则库
    │                                    │
    └── 月度校准方向 ←── 长期记忆更完整 ←─┘
```

### 3.4 AI 的三种工作模式

3.2 定义了 AI 是「载入你灵魂的数字合伙人」——底层始终是同一个 AI，同一份 profile.md。但在不同的 PDCA 周期，AI 需要切换关注点，用不同的视角审视你的状况。这就是三种工作模式的由来：

| 工作模式 | 对应挑战（1.1） | 对应 PDCA 层 | 核心问题 |
|---------|---------------|-------------|---------|
| **Strategy Coach** | 方向迷失 | 战略层（月度） | 我们在往对的方向走吗？ |
| **Project Manager** | 执行低效 | 项目层（周度） | 事情在按计划推进吗？ |
| **Skill Analyzer** | 经验流失 | 能力层（日度） | 今天的经验能沉淀什么？ |

三种模式共享同一个底层（profile.md + 记忆系统 + 规则库 + 技能库），但各自用不同的视角运作：Strategy Coach 用你的价值观做战略判断，Project Manager 用你的工作标准做进度评估，Skill Analyzer 用你的思维方式做知识提炼。

#### Strategy Coach（战略顾问）

**为什么需要这个模式**：日常协作中，AI 和你一样容易陷入细节，失去对大方向的感知。Strategy Coach 模式强制 AI 拉远视角，站在月度/季度的尺度上审视——你忙的这些事，真的在接近目标吗？

**触发时机**：每月 1-3 日

**输入**：
- `Strategy/north-star.md`（长期方向）
- `Strategy/quarterly/`（季度目标）
- `Calendar/Reviews/Weekly/`（近一个月的周报）
- `memory/long_term.md`（你的长期目标和偏好）

**思考框架**：
1. 这些行动真的在接近目标吗？（对齐度检查）
2. 投入产出比最高的是哪 3 件事？（杠杆点识别）
3. 哪些是伪工作？（噪音过滤）
4. 最大的机会成本是什么？（隐藏代价）
5. 如果只能做 1 件事，应该是什么？（本质提炼）

**输出**：`Strategy/reviews/YYYY-MM-review.md`

**喂养复利**：战略判断经验 → 规则库；近期记忆中稳定化的信息 → 压缩到长期记忆

---

#### Project Manager（项目经理）

**为什么需要这个模式**：项目一多，优先级就会混乱，拖延和风险容易被忽视。Project Manager 模式让 AI 以周为单位做全景扫描——哪些在推进、哪些卡住了、时间花得值不值。

**触发时机**：每周五下午

**输入**：
- `Efforts/Projects/Active/`（进行中的项目）
- `Calendar/Days/`（本周每日记录）
- `Strategy/quarterly/`（本季度目标，用于校准优先级）
- `memory/recent_rolling.md`（近期动态）

**思考框架**：
1. 进度追踪：完成了什么？离目标多远？
2. 风险识别：哪些在拖延？为什么？
3. 依赖分析：项目间是否冲突或阻塞？
4. 效能分析：时间分配合理吗？高价值任务占比多少？

**输出**：`Calendar/Reviews/Weekly/YYYY-WXX.md`

**喂养复利**：本周踩的坑 → 规则库；有效的工作流程 → 技能库

---

#### Skill Analyzer（能力分析师）

**为什么需要这个模式**：每天都在做事、学东西，但如果不刻意提炼，经验就像流水一样过去了。Skill Analyzer 模式让 AI 在每天结束时扮演「知识蒸馏器」——从零散经历中萃取可复用的知识。

**触发时机**：每晚复盘时

**输入**：
- `Calendar/Days/` 当日笔记（今天做了什么、学了什么）
- `Atlas/Dots/Skills/_index.md`（已有技能索引，避免重复）
- `Atlas/Maps/`（已有知识地图，寻找关联）

**思考框架**：
1. 从零散经验中提炼可复用知识
2. 识别模式：这个和之前哪个经验相关？
3. 升维思考：这背后的原理是什么？
4. 建议沉淀：应该加到哪个 Dots？还是创建新的？
5. 关联 MOC：这属于哪个知识地图？

**输出**：追加到今日笔记 + 更新 `Atlas/Dots/`

**喂养复利**：今天学会的方法 → 技能库；更新记忆

## 四、成长管理模块

### 4.1 三层 PDCA 循环

#### 为什么是三层？

1.1 提出了三个挑战——方向迷失、执行低效、经验流失。每个挑战对应一个时间尺度和一种迭代节奏：

| 挑战 | 对应层 | 时间尺度 | 核心问题 |
|-----|--------|--------|---------|
| 方向迷失 | 战略层（慢回路） | 月度/季度 | 我在往对的方向走吗？ |
| 执行低效 | 项目层（中回路） | 周度 | 事情在按计划推进吗？ |
| 经验流失 | 能力层（快回路） | 日度 | 今天的经验能沉淀什么？ |

为什么不用一层统一的 PDCA？因为三个挑战的时间尺度完全不同：

- 方向校准不能太频繁（每天质疑战略会焦虑），也不能太稀疏（一年才看一次就晚了）→ 月度
- 项目推进需要中等粒度的检查点，太短看不出趋势，太长来不及调整 → 周度
- 知识积累必须日积月累，断了就断了 → 日度

三层各自运转，但嵌套为一个整体：

```
┌─────────────────────────────────────────────────────────┐
│                     战略层（慢回路）                      │
│           周期：月度/季度 │ 校准方向                      │
│  ┌─────────────────────────────────────────────────┐   │
│  │                  项目层（中回路）                  │   │
│  │         周期：周度 │ 推进目标                      │   │
│  │  ┌─────────────────────────────────────────┐    │   │
│  │  │              能力层（快回路）              │    │   │
│  │  │       周期：日度 │ 积累复利                │    │   │
│  │  └─────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

#### 三层如何相互驱动

三层不是独立运行的，而是形成嵌套循环：

- **战略层 → 项目层**：方向决定行动。季度目标分解为具体项目，确保每个 Active 项目都能回答「这为什么值得做？」
- **项目层 → 能力层**：项目提供场景。做项目的过程产生真实经验，给知识积累提供素材——没有实践的知识是空谈
- **能力层 → 战略层**：数据支撑决策。日度积累的知识和周度的项目数据，构成月度战略复盘的事实基础——没有数据的战略是空想

```
战略层输出方向 ──→ 项目层据此执行 ──→ 能力层在执行中积累
      ↑                                    │
      └──── 积累的数据反哺战略判断 ←────────┘
```

#### PDCA 在每层的含义

PDCA（Plan-Do-Check-Act）在三层中含义不同：

| | Plan（计划） | Do（执行） | Check（检查） | Act（调整） |
|-----|------------|----------|-------------|-----------|
| **战略层** | 设定北极星、年度/季度目标 | 用目标指导项目层 | 月度战略复盘 | 调整方向、砍项目 |
| **项目层** | 从季度目标分解项目和里程碑 | 每周推进任务 | 周度复盘 | 调整优先级和状态 |
| **能力层** | 识别需积累的知识领域 | 每天记录和学习 | 每晚提炼 | 沉淀到知识库 |

每层的 Check 由对应的 AI 工作模式驱动：战略层由 Strategy Coach 驱动，项目层由 Project Manager 驱动，能力层由 Skill Analyzer 驱动（详见 3.4）。Check 环节也是复利机制的主要喂养入口——每次 Check 都产出可复用资产（详见 3.3）。

### 4.2 战略层：校准方向

#### 设计意图

战略层解决的是 1.1 中的第一个挑战——**方向迷失**：忙了一年，回头发现没有实质进展。

传统年度计划和 OKR 的问题不是「计划不好」，而是「写完就忘」——缺少持续的校准机制。PersonalOS 的战略层设计了一套**层层递进的目标体系 + 定期校准机制**，确保长期方向持续发挥作用。

#### 目标体系：从远到近

```
北极星（3-5 年）
   ↓ 分解
年度目标（3-5 个）
   ↓ 聚焦
季度目标（1-3 个）
   ↓ 校准
月度复盘（检查对齐度）
```

每一级只做一件事：

| 文件 | 回答的问题 | 特点 |
|-----|----------|------|
| `north-star.md` | 3-5 年后我想成为什么样的人？ | 很少变动，是一切的起点 |
| `annual-plan.md` | 今年要在哪几个方向取得进展？ | 每年初设定，年中可微调 |
| `quarterly/YYYY-QX.md` | 这个季度最重要的 1-3 件事是什么？ | 每季度重新聚焦 |
| `reviews/YYYY-MM-review.md` | 上个月的行动和方向对齐吗？ | 月度战略复盘，由 Strategy Coach 驱动 |
| `reviews/YYYY-QX-review.md` | 这个季度的战略是否需要重排？ | 季度战略复盘，重估目标和资源 |
| `reviews/YYYY-review.md` | 这一年的方向是否有效？ | 年度战略复盘，输出下一年约束 |

**为什么需要四级？**

- **北极星**提供锚点——当你犹豫时，回看北极星，选择更接近它的那条路
- **年度目标**是大方向——一年能做的事有限，必须取舍
- **季度目标**是焦点——把年度目标拆成可执行的阶段重点，避免「什么都想做」
- **月度复盘**是校准——检查行动是否偏离，避免「写完就忘」

没有北极星，季度目标容易随波逐流；没有季度聚焦，年度目标太大无法行动；没有月度复盘，季度目标写了等于没写。

#### 复盘双轨口径（选择性双轨）

- `Calendar/Reviews/Weekly/`：执行诊断（事实与进度）
- `Calendar/Reviews/Monthly/`：可选运营月报（数据汇总，不下战略结论）
- `Strategy/reviews/`：战略复盘（月/季/年）
- 同一周期不得在两个目录写同一种用途的复盘
- 月度双轨并存时，标题必须显式区分「运营月报」和「战略月复盘」

#### 战略层的 PDCA

```
Plan：年初设定北极星和年度目标 → 季初设定季度目标
 ↓
Do：用季度目标指导项目层（决定做什么项目、砍什么项目）
 ↓
Check：每月 1-3 日，Strategy Coach 读取整月数据，进行战略质疑（详见 3.4）
 ↓
Act：调整季度目标、重新分配精力、砍掉不对齐的项目
```

战略层的 Check 不是「完成了多少」，而是「方向对不对」——Strategy Coach 的核心价值是质疑而非汇报：「你忙的这些事，真的在接近目标吗？」

### 4.3 项目层：推进目标

#### 设计意图

项目层解决的是 1.1 中的第二个挑战——**执行低效**：项目拖延、优先级混乱、任务堆积。

这个问题的根源通常不是「做得不够多」，而是「同时做太多」。人的精力有限，同时推进十个项目等于一个也推不动。项目层的核心设计是**三状态管理**——不是管你做了多少，而是管你放弃了多少。

#### 三状态项目管理

| 状态 | 含义 | 精力投入 | 典型数量 |
|-----|------|---------|---------|
| **Active** | 正在推进，本周有明确行动 | 高 | 3-5 个 |
| **Simmering** | 酝酿中，暂时搁置但保持关注 | 低 | 不限 |
| **Sleeping** | 休眠中，完成或无限期暂停 | 无 | 不限 |

**为什么需要 Simmering？**

传统项目管理只有「进行中」和「已完成」——一个项目要么在做，要么做完了。但现实中存在大量「不做了但又没完全放弃」的项目，它们堆在「进行中」列表里，制造焦虑，占据心智带宽。

Simmering（酝酿中）给了你一个正式的「先放一放」的位置。把项目移到 Simmering 不是放弃，而是有意识地释放精力，同时保持「想到了随时可以捡回来」的灵活性。

**Active 数量的硬约束**：建议不超过 5 个。超过 5 个时，Project Manager 会在周度复盘中提醒你做取舍（详见 3.4）。

#### 项目与战略的对齐

每个 Active 项目都应该能回答：**这个项目服务于哪个季度目标？**

```
Strategy/quarterly/2026-Q1.md
├── 目标1：建立个人品牌
│   ├── Efforts/Projects/Active/blog-writing.md
│   └── Efforts/Projects/Active/newsletter-launch.md
│
└── 目标2：提升技术深度
    └── Efforts/Projects/Active/rust-learning.md
```

如果一个 Active 项目对不上任何季度目标，要么这个项目该移到 Simmering，要么说明季度目标需要调整。Project Manager 在周度复盘中会做这个对齐检查。

#### 项目层的 PDCA

```
Plan：从季度目标分解 Active 项目，设定里程碑和任务
 ↓
Do：每周推进任务，在每日笔记中记录进展
 ↓
Check：每周五，Project Manager 生成周报，检查进度和风险（详见 3.4）
 ↓
Act：调整项目优先级、移动项目状态（Active ↔ Simmering）、重新分配时间
```

项目层的 Check 关注「进度」和「效能」——不只是完成了多少，还要看时间花得值不值。

### 4.4 能力层：积累知识

#### 设计意图

能力层解决的是 1.1 中的第三个挑战——**经验流失**：学过的忘了，踩过的坑再踩一次。

知识积累的挑战不是「记不住」，而是「找不到」和「连不上」——你可能记了一堆笔记，但用的时候想不起来，因为知识是孤立的、碎片化的。能力层用 Ideaverse 的知识管理框架解决这个问题。

#### 知识管理框架（Atlas/）

```
Atlas/
├── Dots/        原子笔记——最小知识单元
│   ├── People/        人物笔记
│   ├── Things/        事物概念
│   ├── Quotes/        引言金句
│   ├── Statements/    观点主张
│   ├── Questions/     问题探索
│   └── Skills/        可复用技能（复利机制的一部分，详见 3.3）
│
├── Maps/        MOC（Map of Content）——主题索引页
│
└── Sources/     信息来源
    ├── Books/         书籍笔记
    ├── Articles/      文章摘录
    ├── Courses/       课程笔记
    └── Clippings/     网页剪藏
```

**三层结构的设计逻辑**：

- **Dots（原子笔记）**：一个笔记只讲一个知识点，足够小才能被灵活引用和组合。「场景化案例比道理更有说服力」就是一个 Dot。
- **Maps（MOC 索引页）**：把相关的 Dots 组织成主题视角。不是文件夹分类，而是「看问题的角度」——同一个 Dot 可以出现在多个 Map 中。
- **Sources（信息来源）**：知识的源头记录。读了一本书、上了一个课程，笔记归在这里，提炼出的知识点变成 Dots。

#### MOC：知识网络的核心机制

MOC（Map of Content）是能力层最关键的设计。它解决的问题是：**知识不应该被锁死在一个分类里**。

举例：你学到一个知识点——「场景化案例比道理更有说服力」。它可以被多个 MOC 引用：

```
Atlas/Dots/Statements/场景化案例原则.md
├── 被 Writing MOC 引用（写作技巧的一部分）
├── 被 Persuasion MOC 引用（说服力技巧的一部分）
└── 被 Storytelling MOC 引用（叙事方法的一部分）
```

同一个知识点在不同上下文中被激活——这就是知识网络的威力。孤立的笔记是死的，被多个 MOC 引用的笔记是活的。

Skill Analyzer 在每晚提炼时正是在做这件事：识别今天的经验和已有知识的关联，建议加到哪个 Dot、关联到哪个 MOC（详见 3.4）。

#### 能力层的 PDCA

```
Plan：根据项目需要和个人兴趣，识别需要积累的知识领域
 ↓
Do：每天在实践和学习中记录（Calendar/Days/）
 ↓
Check：每晚，Skill Analyzer 从今日记录中提炼知识点（详见 3.4）
 ↓
Act：沉淀到 Atlas/Dots/，更新 Maps，建立关联
```

能力层的 Check 关注「提炼」——不是你记了多少笔记，而是有多少经验被转化为可复用的知识。这些知识通过复利机制（详见 3.3）持续增值：技能越积越多，规则越来越准，AI 越用越强。

## 五、系统运转

第四章讲了三层 PDCA 的设计逻辑，这一章换一个视角：**系统在时间中如何运转**。

### 5.1 一个月的运转节奏

把一个月拆开来看，系统是这样运转的：

```
┌─────────────────────────────────────────────────────────────┐
│  Day 1-3（月初）：战略校准                                    │
│  └─ Strategy Coach 读取上月数据 → 战略质疑 → 调整方向         │
├─────────────────────────────────────────────────────────────┤
│  Day 4-7（第一周）：日常执行                                   │
│  └─ 每日记录 → Skill Analyzer 晚间提炼 → 沉淀到 Atlas        │
├─────────────────────────────────────────────────────────────┤
│  Day 7（周五）：周复盘                                        │
│  └─ Project Manager 生成周报 → 识别风险 → 调整项目状态        │
├─────────────────────────────────────────────────────────────┤
│  Day 8-28：重复「日常执行 + 周复盘」循环                       │
├─────────────────────────────────────────────────────────────┤
│  Day 29-30：为下月战略复盘准备                                │
│  └─ 整理数据 → 准备与 Strategy Coach 对话                    │
└─────────────────────────────────────────────────────────────┘
```

**三层循环的嵌套关系**：

| 循环 | 周期 | 触发时机 | AI 角色 | 主要产出 |
|-----|------|---------|---------|---------|
| 快回路 | 日度 | 每晚复盘 | Skill Analyzer | 知识沉淀到 Atlas |
| 中回路 | 周度 | 每周五 | Project Manager | 周报 + 项目调整 |
| 慢回路 | 月度 | 每月 1-3 日 | Strategy Coach | 战略复盘 + 方向调整 |

快回路嵌套在中回路里，中回路嵌套在慢回路里——就像齿轮一样咬合转动。

### 5.2 信息流动

从信息的视角看系统，数据是这样流动的：

```
+ (Inbox)                           # 快速捕获入口（所有临时 md 文件）
    ├─→ Calendar/Days/              # 日常记录 → 每日笔记
    ├─→ Atlas/Sources/Books/        # 书籍笔记 → 来源库
    ├─→ Atlas/Sources/Clippings/    # 剪藏文章 → 来源库
    ├─→ Atlas/Dots/                 # 知识点 → 原子笔记
    └─→ Efforts/Projects/Simmering/ # 灵感想法 → 酝酿项目

Calendar/Days/                      # 每日记录（原始素材）
    ↓ 提炼（Skill Analyzer）
Atlas/Dots/                         # 知识沉淀（可复用资产）
    ↓ 关联
Atlas/Maps/                         # MOC 索引（知识网络）

Calendar/Days/                      # 每日记录
    ↓ 周聚合（Project Manager）
Calendar/Reviews/Weekly/            # 周报（进度和风险）
    ↓ 月聚合（可选）
Calendar/Reviews/Monthly/           # 运营月报（执行汇总，可选）
    ↓ 作为证据输入
Strategy/reviews/                   # 月度战略复盘（Strategy Coach）
    ↓ 校准
Strategy/quarterly/                 # 季度目标（调整后）
    ↓ 季度/年度校准
Strategy/reviews/YYYY-QX-review.md  # 季度战略复盘
Strategy/reviews/YYYY-review.md     # 年度战略复盘
    ↓ 分解
Efforts/Projects/Active/            # 项目执行
    ↓ 输出
Efforts/Works/                      # 作品发布
```

**三条主线**：
- **入口分流**：Inbox → 根据内容类型流向不同目录（AI 提供建议，你确认后执行）
- **纵向（时间聚合）**：日 → 周 → 月，颗粒度逐渐变粗，视角逐渐拉远
- **横向（知识沉淀）**：原始记录 → 知识点 → 知识网络，经验逐渐结构化

#### 月度双轨最小字段（防冲突）

| 文档 | 最少必须包含 | 禁止项 |
|-----|-------------|-------|
| `Calendar/Reviews/Monthly/YYYY-MM.md` | 数据汇总、项目健康度、执行问题、下月执行建议、证据链接（周报） | 不直接下战略结论 |
| `Strategy/reviews/YYYY-MM-review.md` | 输入证据（周报/运营月报）、数据总结、关键质疑、调整动作、目标校准 | 只给口号，不给证据 |

### 5.3 冷启动与渐进式采用

系统不需要一步到位。建议的启动路径：

**第一周：最小可用**
- 只用 `Calendar/Days/` 写每日记录
- 不需要 AI，不需要复盘，先养成记录习惯

**第二周：加入周复盘**
- 周五花 15 分钟回顾本周
- 可以手动写，也可以尝试让 AI 辅助

**第三周：加入知识沉淀**
- 每晚花 5 分钟，从今日记录中提炼 1-2 个知识点
- 开始建立 `Atlas/Dots/`

**第四周：加入战略层**
- 写下你的北极星和季度目标
- 月初尝试一次战略复盘

**持续迭代**
- 根据实际使用调整节奏
- 不好用的环节可以暂时跳过，好用的环节加强

详细的启动指南见 `01_快速开始.md`。

## 六、工具配置

### 6.1 工具分工

| 区域 | 推荐工具 | 为什么 |
|-----|---------|-------|
| **00_Core/** | Cursor / VS Code | 核心配置文件，方便版本管理和 AI 读取 |
| **01_Shared/** | **Obsidian** | 双链、MOC、插件生态，天然适合 PDCA 系统 |
| **02_Human/** | 任意工具 | 私人空间，选你最舒服的工具 |
| **03_AI/** | Claude Code | AI 自主管理，需要文件操作能力 |

工具不是强制的。整个系统基于 Markdown 文件，任何支持 Markdown 的工具都能用。推荐工具只是基于功能定位的建议。

### 6.2 Obsidian 核心插件

01_Shared/ 使用 Obsidian 时，以下插件支撑 PDCA 系统运转：

| 插件 | 用途 | 必要性 |
|-----|------|-------|
| **Periodic Notes** | 自动创建每日/周笔记，PDCA 循环的基础 | 必须 |
| **Templater** | 模板自动填充，统一笔记格式 | 必须 |
| **Dataview** | 动态查询，构建仪表盘和项目视图 | 推荐 |
| **QuickAdd** | 快速创建各类笔记，降低记录阻力 | 推荐 |
| **Calendar** | 日历视图，可视化时间维度 | 可选 |

### 6.3 AI 配置

三个 AI 工作模式（3.4）可以通过以下方式实现：

**方式一：Claude Code / Cursor 等 AI Coding 工具（推荐）**

这是实现 3.2「数字合伙人」理念的最佳方式：
- AI 可以读写本地文件，实现复利机制的自动沉淀（规则库、技能库、记忆更新）
- AI 拥有真正的 03_AI/ 工作空间，可以自主探索、创建工具
- AI 可以根据任务自动切换工作模式，无需手动管理多个 Project

配置方式：在 CLAUDE.md 或项目配置中引用 `00_Core/profile.md` 和 `00_Core/awakening_spell.md`。

**方式二：Claude Projects**

适合不想给 AI 文件写入权限的用户，或作为特定场景的补充：
- 为每个工作模式创建独立的 Claude Project
- 手动上传对应的上下文文件（profile.md + 相关数据）
- 局限：无法自动更新记忆和规则库，复利机制需要手动维护

**方式三：其他 AI 工具**

任何支持长上下文的 AI 都可以适配，核心是把 profile.md 作为底层系统。但要实现完整的复利机制，需要 AI 具备文件读写能力。

详细的 System Prompt 配置见 `04_AI配置完整版.md`。
