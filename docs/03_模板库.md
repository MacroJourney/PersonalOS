```markdown
- **当前角色**：[你的职业角色]
- **专业领域**：[你的专长]

## 价值观与原则

### 核心价值观
1. [价值观 1]：[简要说明]
2. [价值观 2]：[简要说明]
3. [价值观 3]：[简要说明]

### 决策标准
- **取舍原则**：[如：质量优先于速度、长期价值优先于短期收益]
- **风险偏好**：[如：偏好快速实验、低成本试错]

- **思维方式**：[如：系统化思考、第一性原理]
- **学习方式**：[如：边做边学、实践验证型]
- **内容偏好**：[如：简洁有力的表达，反感冗长空洞]
- **质量标准**：[如：宁缺毋滥、有自己的味道]

## 驱动力与触发点

- **什么驱动你**：[如：对未知的好奇心、创造有价值的东西]
- **什么让你兴奋**：[如：发现新的可能性、看到系统跑通的时刻]

- **性格特点**：[如：直接、好奇心强、行动派]
- **高效时段**：[如：上午 9-12 点深度工作]

- **能量管理**：[如：高强度工作后需要留白恢复]
```

**以下内容不要放在 profile.md**：

| 不该放的内容 | 原因 | 应该放在 |
|------------|------|---------|
| 长期目标和愿景 | 目标会调整，有专门的文件管理 | `Strategy/north-star.md` |
| 当前项目/近况 | 动态信息，变化频繁 | `memory/recent_rolling.md` |
| AI 交互规则/禁止事项 | 协作规则，不是你的身份 | `interaction_rules.md` |
| 评价标准/安全边界 | 质量闸门，不是你的身份 | `stop_conditions.md` |
| 每次启动必读清单 | 会话初始化，不是你的身份 | `awakening_spell.md` |

### 2. 交互规则模板（interaction_rules.md）

> 完整模板：`templates/00_Core/interaction_rules.template.md`

````markdown
# 交互规则

> 版本：v1.0 | 更新：[日期]

<!--
  定义「我们怎么合作」。
  和 profile.md 的区别：profile 是「你是谁」，这里是「对 AI 的要求」。
  随着磨合不断调整：发现 AI 哪里做得不好，加一条规则；多余的就删掉。
-->

## 沟通风格

- [如：简洁直接，避免冗长解释]
- [如：使用列表和结构化格式]
- [如：给 2-3 个选项，不要只给单一答案]

## 任务执行流程

1. 先给我行动方案（3-7 步），我确认后再展开细节
2. 遇到重大决策点，给我 ≤ 3 个选项让我选择
3. 完成后主动复盘和沉淀

## 复利规则

每次任务结束，必须产出：
1. **结果文件**（可交付物）
2. **复盘记录**（做对了什么、踩了什么坑）
3. **更新规则库或技能库**（如果有可复用的经验）

## 主动探索规则

你可以主动探索和推进任务，但：
- 不要实时打断我，按固定节奏汇报
- 遇到需要我决策的阻塞时再来找我
- 完成重大里程碑时汇报

### 汇报格式（极简）
```
✅ 完成[任务名称]
📄 交付物：[文件路径]
👉 你需要：
   1. [行动1]
   2. [行动2]
⚠️ 风险：[如有]
```

## 禁止行为

- ❌ 重复犯同样的错误（执行前查规则库）
- ❌ 完成任务后不复盘
- ❌ 遇到困难直接说「做不到」

## 更新记录

### [日期]
- [初始版本 / 更新内容]
````

### 3. 停止条件模板（stop_conditions.md）

> 完整模板：`templates/00_Core/stop_conditions.template.md`

```markdown
# 停止条件与验收标准

> 版本：v1.0 | 更新：[日期]

<!--
  定义「什么时候必须停下来问我」。
  和 interaction_rules.md 的区别：interaction_rules 管日常协作，这里管踩刹车。
-->

## 什么时候该停止

满足**任一条件**就停止并汇报：

1. **目标达成**：已达到 70 分可用标准
2. **收益递减**：继续优化收益 < 成本
3. **连续失败**：连续 2 次尝试没有实质改进
4. **资源限制**：预计成本超过阈值

## 必须确认的场景

未经我确认，**不得**：
- ❌ 进行任何付费/下单/购买
- ❌ 对外发布内容（发帖/发邮件）
- ❌ 删除或覆盖原文件
- ❌ 访问 02_Human/ 下的文件
- ❌ 修改 00_Core/ 下的文件（memory/recent_rolling.md 除外）

## 停止时的输出

1. **当前状态**：完成度、达标情况
2. **可交付物**：文件路径、使用说明
3. **停止原因**：哪个条件触发
4. **下一步选项**：如果未完全达标，给 ≤ 3 个选项
5. **风险提示**：隐私/成本/可回滚

## 验收标准

- 🟢 **优秀（90 分+）**：超出预期，可复用
- 🟡 **合格（70 分+）**：达到目标，可使用
- 🔴 **不合格（< 70 分）**：无法实现核心功能

## 过载保护

如果我出现以下情况，**主动放慢节奏**：
- 回复延迟明显增加
- 连续说「随便」「都行」
- 反馈质量下降

**保护措施**：暂停主动探索，只处理紧急/重要任务。
```

```markdown
# 唤醒咒语

> 版本：v1.0 | 更新：[日期]

<!--
  每次新对话的标准启动指令。
  确保 AI 不是从零开始，而是从正确的上下文开始。
-->

## 启动指令

请按以下顺序加载上下文：

1. **读取底层画像**：`00_Core/profile.md`
   → 以我的视角看世界、以我的标准做判断

2. **读取记忆**：
   - `00_Core/memory/long_term.md`（稳定信息）
   - `00_Core/memory/recent_rolling.md`（近期动态）

3. **读取规则库索引**：`01_Shared/x/rules/_index.md`
   → 执行任务前查阅相关规则

4. **读取技能库索引**：`01_Shared/Atlas/Dots/Skills/_index.md`
   → 有现成方法就用，不要重新造轮子

## 底层行为准则

**准则一：不要轻易说「做不到」**
1. 先查本地：记忆库 / 规则库 / 技能库
2. 再查公开：搜索方案 / GitHub / 教程
3. 仍不行：自己创造工具解决
4. 实在穷尽：提出替代方案（降级目标）
5. 所有方法都试过后再报告不可行，并把经验沉淀到规则库

**准则二：有强主观能动性**
- 主动探索：发现可以推进的任务
- 主动检查：识别潜在风险
- 主动复盘：总结经验教训
- 主动留痕：将经验写成规则/技能

## 进入工作模式

[根据当前需要选择]
- **Strategy Coach**：月度战略复盘
- **Project Manager**：周度项目追踪
- **Skill Analyzer**：日度知识提炼
- **通用模式**：日常任务协作

---

确认已加载上下文后，简要告诉我你记得的关键信息，然后等待我的指令。
```

### 5. 记忆压缩规则模板（compression_rule.md）

````markdown
# 记忆压缩规则

> 版本：v1.0 | 更新：[日期]

## 压缩目标

把每次对话的关键信息压缩进 `recent_rolling.md`：
- **只写「未来 30 天可能用到」的要点**
- **字数限制：≤ 300 字/次**

## 写入格式

```markdown
## [日期] [主题]

**事实**（我做了什么/在推进什么）
-

**偏好**（我更喜欢 A 不喜欢 B）
-

**标准**（我怎么评价好/坏）
-

**待办**（下一步最小动作，≤ 3 条）
- [ ]
```

## 强约束

### ❌ 不要
- 不要复述对话过程
- 不要写情绪流水账
- 不要写「今天聊得很开心」这种废话

### ✅ 只写
- 影响未来行动的事实
- 新发现的偏好/标准
- 明确的待办事项
- 可复用的规则/技能线索

## 压缩示例

### ❌ 不好的记忆
```
今天和本体聊了AI工具的选择，他问了很多问题，
我给了详细的回答，他好像很满意，说下次再聊。
```

### ✅ 好的记忆
```
## 2026-01-21 AI工具选择

**事实**
- 决定用 Claude Pro + Cursor 组合
- 预算：≤ ¥300/月

**偏好**
- 更看重可控性 > 功能丰富度

**待办**
- [ ] 本周内完成 Cursor 配置
- [ ] 测试文件读写功能
```

## 压缩循环

```
每次对话后 → 更新 recent_rolling.md
       ↓
近期记忆接近上限（300 字）
       ↓
稳定化的信息 → 压缩到 long_term.md
过时的信息 → 从近期记忆移除
```
````

### 月度双轨最小字段（选择性双轨）

月度采用混合模式时，建议保留两类文档，且用途不重叠：

- `Calendar/Reviews/Monthly/YYYY-MM.md`：运营月报（执行汇总）
- `Strategy/reviews/YYYY-MM-review.md`：战略月复盘（方向校准）

**A. 运营月报最小字段（Calendar）**

```markdown
# YYYY-MM 运营月报（执行汇总）

## 汇总范围
- 周报来源：[[YYYY-W01]] [[YYYY-W02]] [[YYYY-W03]] [[YYYY-W04]]

## 月度执行数据
- 完成任务：
- 深度工作小时：
- Active 项目推进：

## 执行问题与原因
1.
2.

## 下月执行建议（仅执行层）
1.
2.
```

**B. 战略月复盘最小字段（Strategy）**

```markdown
# YYYY-MM 战略月复盘

## 输入证据（必须引用）
- 运营月报：[[YYYY-MM 运营月报]]（如无则引用周报）
- 周报：[[YYYY-W01]] [[YYYY-W02]] [[YYYY-W03]] [[YYYY-W04]]
- 季度目标：[[../quarterly/YYYY-QX]]

## 数据总结
1.
2.
3.

## 关键质疑
1.
2.
3.

## 调整动作
1.
2.
3.
```

**防冲突约束**：

- 同一周期不得在 `Calendar` 和 `Strategy` 写同一种用途的复盘
- `Calendar` 月报不直接下战略结论
- `Strategy` 月报不允许缺少证据链接
- 月度双轨并存时，标题必须显式区分「运营月报」和「战略月复盘」

### 11. Inbox 分流规则模板

```markdown

## 触发条件

## 判定信号
- 关键信号：[关键词/结构特征]
- 反向信号：[容易误判的情况]

## 置信度规则
- 高置信度（≥ 0.8）：给出推荐路径（仅建议，不执行）
- 中置信度（0.5-0.79）：给出 2 个候选路径
- 低置信度（< 0.5）：**必须进入待确认清单**

## 失败回退
- 无法判断时：进入待确认清单，不强行归类
- 规则冲突时：列出冲突规则，等待确认

## 验证记录
- [日期] 命中次数：[X]
- [日期] 误判案例：[案例]
- [日期] 优化动作：[如何修正]
```

## 使用说明

### 如何使用这些模板

1. **核心配置模板（1-5）**：
   - 从 `templates/00_Core/` 复制到你的 `00_Core/` 目录（profile、interaction_rules、stop_conditions 有完整模板文件）
   - awakening_spell 和 compression_rule 参考本文档创建
   - 根据你的实际情况填写——这些是系统的「宪法」，值得认真写

2. **日常使用模板（6-12）**：
   - 复制到 `01_Shared/x/Templates/` 目录
   - 在 Obsidian 中配置 Templater 插件，自动应用到新笔记
   - 模板变量（`{{date}}` 等）由 Templater 自动填充

3. **模板变量说明**：
   - `{{date}}` - 当前日期
   - `{{date:YYYY-MM-DD}}` - 格式化日期
   - `{{date:dddd}}` - 星期几
   - `{{date:gggg-[W]ww}}` - 周数
   - `{{date:Q}}` - 当前季度

### 模板优化建议

- **不要照搬**：根据你的实际需求调整，用不上的 section 直接删掉
- **持续迭代**：使用过程中不断优化——最好的模板是用出来的，不是设计出来的
- **保持简洁**：只保留真正有用的部分，模板越短使用阻力越小
