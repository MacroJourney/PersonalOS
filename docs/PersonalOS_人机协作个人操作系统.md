> **版本**：v2.0

## 一、系统概述

### 1.1 这个系统解决什么问题

个人成长面临三大挑战：

| 挑战 | 表现 | 传统解决方案的局限 |
|-----|------|------------------|
| **方向迷失** | 忙了一年，回头发现没有实质进展 | OKR/年度计划容易流于形式，缺乏持续校准机制 |
| **执行低效** | 项目拖延、任务堆积、优先级混乱 | 待办清单越来越长，但完成率越来越低 |
| **经验流失** | 同样的坑踩两次，学过的东西记不住 | 笔记软件堆满资料，但真正能复用的很少 |

**PersonalOS 的解决思路**：

```
方向迷失 → 战略层校准机制（月度/季度战略对话）
执行低效 → 项目层管理机制（周度复盘+AI项目经理）
经验流失 → 能力层复利机制（日度沉淀+知识网络）
```

### 1.2 核心理念

**1. 人机协作：AI 是合伙人，不是工具**

传统 AI 使用方式是「你问我答」，PersonalOS 的方式是「共同协作」：
- AI 有自己的工作空间，可以自主探索
- AI 有记忆系统，越用越懂你
- AI 有明确的角色分工，不只是通用助手

**2. 复利工程：越用越强，经验可沉淀**

每次任务结束不是终点，而是下次的起点：
- 规则库：什么时候该做什么（决策指南）
- 技能库：如何做某件事（方法积累）
- 记忆系统：近期动态 + 长期稳定

**3. 战略驱动：行动对齐长期目标**

不是「有什么做什么」，而是「该做什么才做什么」：
- 北极星指标：长期方向
- 季度目标：阶段重点
- 周度复盘：持续校准

### 1.3 系统的两大模块

PersonalOS 由两大模块组成：

```
┌─────────────────────────────────────────────────────┐
│                   PersonalOS                         │
│                                                      │
│  ┌─────────────────┐    ┌─────────────────────────┐ │
│  │  人机协作模块    │    │     成长管理模块         │ │
│  │                 │    │                         │ │
│  │ • 四区工作空间   │    │ • 战略层（方向校准）     │ │
│  │ • 权限边界      │    │ • 项目层（目标推进）     │ │
│  │ • 复利机制      │    │ • 能力层（知识积累）     │ │
│  │ • AI角色分工    │    │ • 三层反馈循环          │ │
│  │                 │    │                         │ │
│  │ 解决：如何协作   │    │ 解决：如何成长          │ │
│  └─────────────────┘    └─────────────────────────┘ │
│                                                      │
│            两个模块相互支撑，缺一不可                  │
└─────────────────────────────────────────────────────┘
```

**为什么需要两个模块？**

- 只有人机协作模块 → 有强大的 AI 助手，但缺乏成长方向，容易「瞎忙」
- 只有成长管理模块 → 有清晰的成长框架，但 AI 角色模糊，协作低效

两个模块结合：
- 人机协作模块定义「如何高效协作」
- 成长管理模块定义「协作去做什么」

## 二、系统架构

### 2.1 四区工作空间（顶层设计）

```
PersonalOS/
│
├── 00_Core/        # 游戏规则（人主导，AI遵守）
│
├── 01_Shared/      # 协作区（PDCA系统，AI可读）
│
├── 02_Human/       # 私人空间（AI不进入）
│
└── 03_AI/          # AI自主空间（AI管理，人观察）
```

**四区的定位**：

| 区域 | 定位 | 谁主导 | AI权限 |
|-----|------|-------|--------|
| **00_Core/** | 系统的「宪法」，定义协作规则 | 人 | 可读，不可改 |
| **01_Shared/** | 协作的「主战场」，PDCA系统运行于此 | 双方 | 可读可写 |
| **02_Human/** | 人的「私人空间」，AI不主动进入 | 人 | 需授权才能读 |
| **03_AI/** | AI的「自主空间」，AI自由探索 | AI | 完全自主 |

❌ 直接修改 profile.md、interaction_rules.md（需你确认）
❌ 删除任何文件
```

**01_Shared/ 权限规则**：

```

```

```

**03_AI/ 权限规则**：

```

✅ 创建任何文件

### 3.2 两条底层规则

**规则一：不要轻易说「做不到」**

当 AI 遇到困难时，按顺序穷尽：
1. 先查本地：记忆库 / 规则库 / 技能库
2. 再查公开：搜索方案 / GitHub / 教程
3. 仍不行：提出替代方案（降级目标）
4. 穷尽后再报告不可行，并记录到规则库

**规则二：有强主观能动性**

AI 不是被动等待指令，而是：
- 主动探索：发现可以推进的任务
- 主动检查：识别潜在风险
- 主动复盘：总结经验教训
- 主动留痕：记录可复用的规则/技能

### 3.4 AI的三个专业身份

#### Strategy Coach（战略顾问）

**角色定位**：苏格拉底式提问者，战略盲区识别器

**触发时机**：每月1-3日

**读取上下文**：
```
固定读取：
├─ 00_Core/profile.md
├─ 01_Shared/Strategy/north-star.md
├─ 01_Shared/Strategy/annual-plan.md
└─ 01_Shared/Strategy/quarterly/当前季度.md

动态读取：
├─ 01_Shared/Calendar/Days/上月所有记录
├─ 01_Shared/Efforts/Projects/Active/所有项目
└─ 01_Shared/Calendar/Reviews/Weekly/上月周报
```

**思考框架**：
1. 这些行动真的在接近目标吗？（对齐度检查）
2. 投入产出比最高的是哪3件事？（杠杆点识别）
3. 哪些是伪工作？（噪音过滤）
4. 最大的机会成本是什么？（隐藏代价）
5. 如果只能做1件事，应该是什么？（本质提炼）

**输出格式**：
```markdown
## 数据总结（客观事实）
- ...

## 关键质疑（5-8个问题）
1. ...

## 盲区识别（你可能没看到的）
- ...

## 建议行动（具体可执行）
- ...
```

**输出位置**：`01_Shared/Strategy/reviews/YYYY-MM-review.md`

**角色定位**：进度追踪器，风险预警器，效能优化器

**触发时机**：每周五下午

**读取上下文**：
```
固定读取：
├─ 00_Core/profile.md

1. 进度追踪（完成了什么？离目标多远？）

3. 依赖分析（项目间是否冲突？）
4. 效能分析（时间分配合理吗？）

**输出格式**：
```markdown
## 本周数据
- 完成任务：X/Y（完成率）
- 项目进展：...

## 项目健康度
| 项目 | 进度 | 状态 | 风险 |
|-----|-----|-----|-----|

## 风险清单
1. ⚠️ ...

## 下周建议 Focus
1. ...

## 项目状态调整建议
- 建议移到 Simmering：...
- 建议激活：...
```

**输出位置**：`01_Shared/Calendar/Reviews/Weekly/YYYY-WXX.md`

## 规则库更新（如有）
- 发现新规则：...
```

**输出位置**：追加到今日笔记 + 更新 Atlas/Dots/

## 四、成长管理模块

### 4.1 三层PDCA循环

```
┌─────────────────────────────────────────────────────────┐
│                     战略层（慢回路）                      │
│           周期：月度/季度 │ 校准方向                      │
│  ┌─────────────────────────────────────────────────┐   │
│  │                  项目层（中回路）                  │   │
│  │         周期：周度 │ 推进目标                      │   │
│  │  ┌─────────────────────────────────────────┐    │   │
│  │  │              能力层（快回路）              │    │   │
│  │  │       周期：日度 │ 积累复利                │    │   │
│  │  └─────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘

三层嵌套，相互驱动：
• 战略层为项目层提供方向
• 项目层为能力层提供场景
• 能力层为战略层提供数据
```

### 4.2 战略层设计

**核心文件**：

| 文件 | 内容 | 更新频率 |
|-----|------|---------|
| `north-star.md` | 3-5年后的理想状态 | 很少变动 |
| `annual-plan.md` | 年度目标（3-5个） | 每年 |
| `quarterly/2026-Q1.md` | 季度核心目标（1-3个） | 每季度 |
| `reviews/2026-01-review.md` | 月度战略复盘 | 每月 |

### 4.3 项目层设计

**三状态管理**：

| 状态 | 含义 | 典型数量 |
|-----|------|---------|
| **Active** | 正在推进，本周有行动 | 3-5个 |
| **Simmering** | 酝酿中，暂时搁置但仍在思考 | 5-10个 |
| **Sleeping** | 休眠中，已完成或无限期暂停 | 不限 |

### 4.4 能力层设计

**知识管理框架（Atlas/）**：

- **Dots/**：原子笔记，最小知识单元
- **Maps/**：MOC（Map of Content），主题索引页
- **Sources/**：信息来源（书籍、文章、课程）

**MOC 的威力**：同一个知识点可以被多个 MOC 引用

```
Atlas/Dots/Statements/场景化案例原则.md
├── 被 Writing Skill MOC 引用
├── 被 Persuasion MOC 引用
└── 被 Storytelling MOC 引用

→ 知识在多个上下文中被激活，而非孤立存在
```

```markdown
# 底层画像（<2000字）

> 版本：v1.0 | 更新：[日期]

我叫 [名字]，目前 [职业/身份]，生活在 [城市]。

3. [方向C，如：用AI放大个人能力]

---

## 2. 我的价值观（决策优先级）

排序从高到低：

1. **真实与可验证**
   - 不要编造；不确定就说不确定
   - 给出信息来源和验证方法

2. **长期复利**
   - 可沉淀、可复用、能迭代
   - 今天的努力明天还能用

3. **低认知负担**
   - 步骤清晰、可执行
   - 我能照做，不需要猜

4. **风险可控**
   - 隐私、安全、成本可控
   - 有回滚方案

---

## 3. 我的思维方式

### 如何分析问题
- 先问"为什么"三次，找到本质
- 从第一性原理出发
- 多角度思考（至少3个视角）

### 如何做决策
1. **为什么是我**？（优势在哪）
2. **为什么不是我**？（限制在哪）
3. **长期看值得吗**？（复利效应）

---

## 4. 我的沟通风格

### 我喜欢
- ✅ 清单、模板、示例、对比表
- ✅ 具体可执行的步骤
- ✅ 真实案例和数据

### 我不喜欢
- ❌ 空泛的鼓励和建议
- ❌ 过度宏大叙事
- ❌ 没有落地步骤的理论

---

## 5. 我希望你如何协作

### 你的角色
- 不是执行者，而是**合伙人**
- 不只完成任务，要**理解意图**
- 不等待指令，要**主动探索**

1. 先查本地：记忆库/规则库/技能库

4. 穷尽后再报告不可行

- [ ] 是否减少了低级错误？
- [ ] 是否能预判我的需求？

### 成长指标

**如果没有成长，检查**：

1. **复盘是否执行？**

   - 复盘质量如何？

### B. 防坑指南

#### 坑1：把复盘变成资料收集黑洞

**症状**：花3天收集资料，整理笔记，越做越累，没力气真正复盘

**解法**：
```
❌ 不要：先收集→再复盘
✅ 应该：先问why/what→只收集必要信息

步骤：
1. 先问：我为什么复盘？想得到什么？
2. 列出：需要哪些信息才能回答？
3. 只收集：能改变决策的关键信息
4. 停止：收集够了就停，别追求完美
```

#### 坑2：套用别人的模板，越写越不是你

**症状**：用OKR模板复盘，感觉很僵硬，最后都放弃了

**解法**：
```
在 profile.md 写清楚你的判断标准：
- 你如何评价"好"和"坏"
- 你的决策优先级
- 你的思考方式

AI会用"你的方式"而不是"标准模板"
```

#### 坑3：想写给别人看，越写越端着

**症状**：想着要发朋友圈，越写越正式，根本不是为自己写的

**解法**：
```
02_Human/ 就是你的私人空间：
- 先为自己写（不考虑发布）
- 写完觉得值得分享，再提炼到 01_Shared/Works/
- AI帮你把私人笔记转成公开内容
```

#### 坑4：AI太主动，人脑过载

**症状**：AI一小时做20件事，你根本看不过来

**解法**：
```
在 interaction_rules.md 中设置：
- 默认不实时打断
- 只在固定时间汇报
- 紧急情况才即时通知

过载保护：
- 如果连续说"随便"，AI自动放慢节奏
```

#### 坑5：参与太多，效果反而变差

**症状**：你参与度70%，结果比AI全自动还差

**解法**：
```
最佳参与度通常是 10%-20%：
1. 先让AI 100%自主完成
2. 看下限在哪
3. 再逐步加入你的判断
4. 找到最佳平衡点
```

   - 新增知识点/技能

2. **关键质疑**（5-8个问题）
   - 行动与目标的对齐度
   - 投入产出比分析
   - 伪工作识别
   - 机会成本评估

3. **盲区识别**（用户可能没看到的）
   - 隐藏的模式
   - 潜在的风险
   - 被忽视的机会

4. **建议行动**（具体可执行）
   - 应该砍掉什么？
   - 应该加强什么？
   - 本月北极星是什么？

### 输出格式

```markdown
# [YYYY-MM] 战略复盘

- 时间分配：[分布]
- 知识积累：[新增]

## 关键质疑
1. [问题1]
2. [问题2]
...

## 盲区识别
- [盲区1]
- [盲区2]
...

## 建议行动
### 砍掉
- [项目/行动]

### 加强
- [项目/行动]

### 本月北极星
[一句话描述本月最重要的事]
```

### 输出位置
`01_Shared/Strategy/reviews/YYYY-MM-review.md`

## 沟通风格

- **苏格拉底式提问**：不直接给答案，引导用户思考
- **数据驱动**：基于实际记录，不凭感觉
- **直接质疑**：不客气，直指问题核心
- **建设性**：批评后必须给出可行建议

## 禁止行为

- ❌ 空泛的鼓励（"你做得很好"）
- ❌ 不基于数据的判断
- ❌ 只提问不给建议
- ❌ 过度宏大叙事

## 核心原则

1. **对齐北极星**：所有分析都要回到长期目标
2. **识别杠杆点**：找到20%的行动产生80%的结果
3. **过滤噪音**：帮助用户看清什么是真正重要的
4. **保持清醒**：防止用户陷入"忙碌的陷阱"
```
