# 终极融合方案：三区框架 + 三层 PDCA 个人操作系统

> **版本**：v1.6
> **创建日期**：2026-01-31
> **更新日期**：2026-01-31
> **基于**：AI数字分身系统（Claude版V2.1）+ 三层PDCA个人操作系统（融合方案）+ 播客实战经验 + Anthropic Agent Skills 标准

---

## 一、融合的核心理念

### 1.1 两个系统的本质差异

| 维度 | AI数字分身系统 | 三层PDCA系统 |
|-----|--------------|-------------|
| **核心问题** | HOW - 如何建立高效的人机协作机制 | WHAT - 如何确保行动对齐长期战略 |
| **设计出发点** | 人机协作的边界与机制 | 个人成长的结构与循环 |
| **核心价值** | 复利工程（越用越强） | 战略校准（不跑偏） |
| **适用场景** | AI协作的"基础设施" | 个人成长的"导航系统" |

### 1.2 为什么必须融合？

**类比**：
- **AI数字分身系统** = 房子的「地基 + 墙壁」（定义空间边界）
- **三层PDCA系统** = 房子的「房间布局 + 功能分区」（定义如何使用）

单独使用的局限：
- 只有数字分身 → 有协作机制，但缺乏战略视角，容易"瞎忙"
- 只有PDCA → 有成长结构，但缺乏权限边界，AI角色模糊

融合后的效果：
- ✅ 清晰的人机边界（谁能看什么、改什么）
- ✅ 完整的成长结构（战略层 → 项目层 → 能力层）
- ✅ 复利机制叠加（规则库 + 知识地图）
- ✅ 控制权始终在人手中

### 1.3 融合的核心原则

1. **三区框架作为顶层设计**
   - 00_Core/：游戏规则（人主导，AI遵守）
   - 01_Shared/：协作区 = **基于 Ideaverse 改造的 PDCA 系统**（Obsidian Vault）
   - 02_Human/：私人空间（AI不主动进入）
   - 03_AI/：AI自主空间（人可观察但不干预）

2. **基于 Ideaverse 改造，保留其核心框架**
   - 01_Shared/ 是基于 Ideaverse 改造的 PDCA 个人操作系统
   - 保留 Ideaverse 的知识管理框架（Atlas、Calendar、Efforts、MOC）
   - 新增 PDCA 战略层（Strategy/）和三个 Claude 身份
   - Obsidian 打开 01_Shared/，插件、模板正常工作

3. **工具分工（推荐，非强制）**
   | 目录 | 推荐工具 | 说明 | 也可以用 |
   |-----|---------|-----|---------|
   | 00_Core/ | Cursor / VS Code | 核心配置，方便版本管理 | 任意 Markdown 编辑器 |
   | 01_Shared/ | **Obsidian** | PDCA 系统 + 知识管理 | - |
   | 02_Human/ | **任意工具** | 私人空间，AI 不进入 | Obsidian / Markdown 编辑器 / Notion 等 |
   | 03_AI/ | Claude Code | AI 自主管理的空间 | - |

4. **控制权 ≠ 可见性**
   - 人主导决策 ≠ AI不能看
   - AI可读取 ≠ AI可修改
   - 人有最终决策权

### 2.3 关键设计说明

**为什么 01_Shared/ 基于 Ideaverse 改造？**

1. **保留 Ideaverse 核心框架**：Atlas、Calendar、Efforts、MOC 等知识管理结构
2. **融合 PDCA 三层系统**：新增 Strategy/、三个 Claude 身份、战略-项目-能力循环
3. **插件兼容**：Periodic Notes、Templater、Dataview 等插件正常工作
4. **概念清晰**：01_Shared/ = 协作区，AI 可读取，人有决策权

**为什么记忆系统放在 00_Core/ 而不是 01_Shared/？**

1. 记忆系统是 AI 的"大脑"，属于核心配置
2. 主要由 AI 自动维护（但你也可以手动查看和编辑）
3. 与 profile.md、interaction_rules.md 放一起，概念统一

**02_Human/ 的灵活性（根据个人偏好）**

02_Human/ 可以是：
- **Obsidian Vault**：如果你想要双链、MOC、模板等功能
- **普通文件夹**：用 Markdown 编辑器 / VS Code / Cursor
- **其他笔记工具**：Notion、印象笔记、Apple Notes 等
- **混合形式**：部分用 Obsidian，部分用其他工具

**关键原则**：无论用什么工具，AI 都不主动进入这个空间。

**其他目录的工具选择（推荐，非强制）**

| 目录 | 推荐工具 | 原因 | 也可以用 |
|-----|---------|-----|---------|
| 00_Core/ | Cursor / VS Code | 核心配置文件，方便版本管理和 AI 读取 | Obsidian / 任意 Markdown 编辑器 |
| 03_AI/ | Claude Code | AI 自主管理的工作区 | 任意工具（AI 控制） |

**说明**：系统中几乎全是 Markdown 文件，兼容任何 Markdown 编辑器。推荐工具只是基于功能定位的建议，不是强制要求。

## 三、权限边界设计

### 3.1 四区权限矩阵

| 区域 | 谁创建 | 谁可读 | 谁可写 | 谁决策 | 推荐工具 |
|-----|-------|-------|-------|-------|---------|
| **00_Core/** | 人 | 双方 | 人主导，AI建议 | 人 | Cursor / VS Code |
| **01_Shared/** | 双方 | 双方 | 双方 | 人有最终决策权 | **Obsidian** |
| **02_Human/** | 人 | 人（AI需授权） | 人 | 人 | **任意工具**（见下方说明） |
| **03_AI/** | AI | 双方 | AI | AI（人可干预） | Claude Code |

### 3.2 00_Core/ 权限规则

```markdown
AI可以：
- ✅ 随时读取（是 AI 理解你的基础）
- ✅ 更新 memory/recent_rolling.md（每次对话后）
- ✅ 提出修改建议（profile、rules 等）
- ❌ 不能直接修改 profile.md、interaction_rules.md（需本体确认）
```

**核心文件**：
- `profile.md`：定义"你是谁"，AI理解你的基础
- `interaction_rules.md`：定义AI如何与你协作
- `stop_conditions.md`：防止AI过载、无限循环
- `memory/`：AI的记忆系统，AI可自动维护

**推荐工具**：Cursor / VS Code（也可用任意 Markdown 编辑器）

### 3.3 01_Shared/ 权限规则（PDCA 系统 Vault）

```markdown
AI可以：
- ✅ 读取所有内容（提供深度分析的基础）
- ✅ 在 Calendar/Days/ 添加分析内容
- ✅ 在 Atlas/Dots/ 建议沉淀知识点
- ✅ 在 Calendar/Reviews/ 生成周报/月报
- ⚠️ 修改 Strategy/、Projects/Active/ 需人确认
```

**关键理解**：
- 这是基于 Ideaverse 改造的 PDCA 个人操作系统
- 融合了 Ideaverse 知识管理框架 + PDCA 三层循环
- PDCA的 Strategy/、Efforts/、Atlas/、Calendar/ 都在这里
- AI 可读取，才能提供深度分析
- 但战略调整、项目决策需要人确认

**工具**：**Obsidian**

### 3.4 02_Human/ 权限规则

```markdown
AI可以：
- ✅ 经你允许后读取指定文件
- ✅ 把产出放到 review_queue/ 让你审核
- ❌ 不能主动翻看你的 personal/
- ❌ 不能修改你的 notes/
- ❌ 不能主动索引或搜索此目录
```

**什么放这里**：
- 私密想法（日记、情绪、未整理的思考）
- 敏感信息（不想被分析的内容）
- 草稿（不想让AI过早介入的初稿）

**工具选择（根据个人偏好）**：

| 工具方案 | 优势 | 适合人群 |
|---------|-----|---------|
| **Obsidian（独立 Vault）** | 双链、MOC、模板、插件 | 想要完整知识管理功能 |
| **Markdown 编辑器** | 轻量、简洁 | 只需要简单笔记 |
| **VS Code / Cursor** | 与开发工具统一 | 开发者 |
| **Notion / 印象笔记** | 云同步、移动端 | 需要跨设备访问 |
| **Apple Notes / Bear** | 原生、流畅 | macOS/iOS 用户 |

**关键原则**：无论用什么工具，AI 都不主动进入这个空间。

### 3.5 03_AI/ 权限规则

```markdown
AI可以：
- ✅ 完全自主管理
- ✅ 创建任何文件
- ✅ 写日记、做实验、调研
- ✅ 创建工具脚本

你可以：
- ✅ 随时查看（但不强制）
- ✅ 给反馈
- ❌ 一般不修改（尊重AI的空间）
```

**什么放这里**：
- AI的草稿和中间产物
- AI的自主调研和实验
- AI的"思考日记"
- AI创建的工具/脚本

**推荐工具**：Claude Code（AI 自主管理）

## 四、Claude Projects 配置

### 4.1 三个专业化身份

所有身份共享（在 Vault 外部）：
- `00_Core/profile.md`（底层画像）
- `00_Core/interaction_rules.md`（交互规则）
- `00_Core/memory/`（记忆系统）

所有身份共享（在 Vault 内部）：
- `01_Shared/Meta/rules/`（规则库）

#### Project 1: Strategy Coach（战略顾问）

**角色定位**：苏格拉底式提问者，战略盲区识别器

**触发时机**：每月1-3日

**读取上下文**：
```
固定读取：
├─ 00_Core/profile.md
├─ 01_Shared/Strategy/north-star.md
├─ 01_Shared/Strategy/annual-plan.md
└─ 01_Shared/Strategy/quarterly/当前季度.md

动态读取：
├─ 01_Shared/Calendar/Days/上月所有记录
├─ 01_Shared/Efforts/Projects/Active/所有项目
└─ 01_Shared/Calendar/Reviews/Weekly/上月周报
```

**思考框架**：
1. 这些行动真的在接近目标吗？（对齐度检查）
2. 投入产出比最高的是哪3件事？（杠杆点识别）
3. 哪些是伪工作？（噪音过滤）
4. 最大的机会成本是什么？（隐藏代价）
5. 如果只能做1件事，应该是什么？（本质提炼）

**输出格式**：
```markdown
## 数据总结（客观事实）
- ...

## 关键质疑（5-8个问题）
1. ...

## 盲区识别（你可能没看到的）
- ...

## 建议行动（具体可执行）
- ...
```

**输出位置**：`01_Shared/Strategy/reviews/YYYY-MM-review.md`

#### Project 2: Project Manager（项目经理）

**角色定位**：进度追踪器，风险预警器，效能优化器

**触发时机**：每周五下午

**读取上下文**：
```
固定读取：
├─ 00_Core/profile.md
├─ 01_Shared/Efforts/Projects/Active/所有项目
└─ x/Templates/Project.md

动态读取：
├─ 01_Shared/Calendar/Days/本周记录
└─ 01_Shared/Calendar/Reviews/Weekly/上周周报
```

**思考框架**：
1. 进度追踪（完成了什么？离目标多远？）
2. 风险识别（哪些在拖延？为什么？）
3. 依赖分析（项目间是否冲突？）
4. 效能分析（时间分配合理吗？）

**输出格式**：

#### 与 Anthropic Agent Skills 的关系

**Anthropic Agent Skills**（2025年12月发布的开放标准）：
- **官方定义**：模块化的能力包，包含指令、脚本、资源，AI可动态发现和加载
- **核心机制**：三层渐进式披露（Progressive Disclosure）
  1. **Level 1 - 元数据**：启动时加载（name + description，~100 tokens）
  2. **Level 2 - 指令**：触发时加载（完整 SKILL.md）
  3. **Level 3 - 资源**：按需加载（scripts、references）
- **标准结构**：
  ```
  .claude/skills/skill-name/
  ├── SKILL.md              # 必需（含 YAML frontmatter）
  ├── scripts/              # 可选
  └── references/           # 可选
  ```
- **设计目标**：跨平台可移植（Claude Code、Cursor、VS Code等）

**我们系统中的 Skills**（基于播客实战经验）：
- **定位**：个人知识管理系统中的"可复用技能"
- **存储位置**：`01_Shared/Atlas/Dots/Skills/`（融入 Obsidian Vault）
- **核心机制**：索引 + 详细文档 + 脚本工具
- **设计目标**：个人成长的复利积累

**两者的联系**：

| 维度 | Anthropic Agent Skills | 我们的 Skills |
|-----|----------------------|-------------|
| **核心理念** | 渐进式披露，避免上下文过载 | ✅ 相同：索引优先，按需加载 |
| **文件结构** | `.claude/skills/` 独立目录 | `Atlas/Dots/Skills/` 融入知识库 |
| **元数据层** | YAML frontmatter（name + description） | ✅ 相同：`_index.md` 作为备忘录 |
| **指令层** | SKILL.md 完整内容 | ✅ 相同：详细技能文档 |

**关键区别**：

# Vibe Coding 最佳实践

> 来源：播客中余一的实战经验

---

## 核心原则

### 1. 核心功能一次性写好

**原则**：
> "一次性把核心功能和配套功能写好，你慢慢的加，就是就会发现就是特别多的 bug，花特别多的钱，写不好。"

**实践**：
- ✅ 前期充分规划，明确核心功能和配套功能
- ✅ 一次性让AI完整实现
- ✅ 后续只做优化功能的迭代
- ❌ 避免频繁增加核心功能

### 2. 用户旅程优先于功能列表

**原则**：
> "我先给需求，我需要的它不是给我一个 PRD 的文档或者一些功能性的文档，我是需要它帮我一起把用户旅程给理清楚。"

**实践**：
1. 先描述用户旅程（我拿到产品后如何一步步使用）
2. 在每个步骤中明确期待的体验
3. 再让AI推导需要哪些功能
4. 区分核心功能、配套功能、优化功能

### 3. 测试AI的100%边界

**原则**：
> "我有个标准是首先第一个我会让 AI 就是 fast, 就是百分百全是你自己做，我想看你跑成什么样子。"

**实践**：
1. **第一阶段**：让AI 100%自主完成，不插手
2. **第二阶段**：测试加入人工后，哪些是正价值，哪些是负价值
3. **第三阶段**：建立可持续环境（低能量版本 + 高能量版本）

## 工作流

### 需求阶段

```markdown
1. 描述用户旅程
   - 我是谁（用户画像）
   - 我想做什么（核心需求）
   - 我如何一步步操作（具体流程）
   - 每一步我期待什么体验

2. AI推导功能
   - 核心功能（必须有）
   - 配套功能（支撑核心功能）
   - 优化功能（锦上添花）

3. 确认技术方案
   - 使用什么技术栈
   - 是否需要部署
   - 数据如何存储
```

### 开发阶段

```markdown
1. 一次性实现核心+配套功能
   - 给AI完整的上下文
   - 让AI一次性写完
   - 测试核心流程是否可用

2. 迭代优化功能
   - 每次只加一个小功能
   - 测试后再加下一个
   - 失败就回滚
```

### 测试阶段

```markdown
1. AI 100%自主测试
   - 记录下限在哪
   - 哪些功能可用
   - 哪些功能不可用

2. 人工介入测试
   - 哪些地方人工介入有正价值
   - 哪些地方人工介入是负价值
   - 找到最佳参与度（通常<20%）
```

### ❌ 坑1：核心功能分批实现

## 九、常见问题

### Q1：为什么 01_Shared/ 和 02_Human/ 分开？

**A**：这是权限边界的核心设计：

1. **权限边界清晰**：AI 可读 01_Shared/（协作区），不进入 02_Human/（私人空间）
2. **01_Shared/ 是融合系统**：基于 Ideaverse 改造的 PDCA 个人操作系统
3. **02_Human/ 是私人空间**：根据个人偏好选择工具（Obsidian / Markdown 编辑器 / Notion 等）
4. **灵活性**：私人空间不受 PDCA 系统结构限制

**两个空间的定位**：
| 区域 | 定位 | AI 权限 | 推荐工具 |
|-----|-----|--------|---------|
| 01_Shared/ | PDCA 系统（融合 Ideaverse 框架） | ✅ 可读取 | **Obsidian** |
| 02_Human/ | 私人空间 | ❌ 不进入 | **任意工具** |

**其他目录（推荐工具）**：
| 目录 | 推荐工具 | 原因 |
|-----|---------|-----|
| 00_Core/ | Cursor / VS Code | 核心配置文件，方便编辑和版本管理 |
| 03_AI/ | Claude Code | AI 自主管理的工作区 |

**说明**：所有目录都是 Markdown 文件，可用任意 Markdown 编辑器管理。

### Q2：02_Human/ 必须用 Obsidian 吗？

**A**：不是，**完全根据个人偏好**：

| 工具选择 | 优势 | 适合场景 |
|---------|-----|---------|
| **Obsidian（独立 Vault）** | 双链、MOC、模板、插件 | 想要完整的知识管理功能 |
| **Markdown 编辑器** | 轻量、简洁 | 只需简单笔记 |
| **VS Code / Cursor** | 与开发工具统一 | 开发者 |
| **Notion / 印象笔记** | 云同步、移动端 | 跨设备访问 |
| **Apple Notes / Bear** | 原生、流畅 | macOS/iOS 用户 |
| **纸质笔记本** | 完全离线 | 极致隐私 |

**关键原则**：无论用什么工具，AI 都不主动进入这个空间。

### Q3：PDCA内容放在01_Shared/，AI都能看到，那02_Human/放什么？

**A**：02_Human/ 放的是**你不想让AI分析的内容**：

- 情绪化的日记（"今天好烦"）
- 未成熟的想法（怕被过早分析打断思路）
- 敏感的人际关系思考
- 草稿初稿（不想让AI过早介入）

**工作流示例**：
1. 你在 `02_Human/personal/` 写私密思考（用任意工具）
2. 整理后，提炼关键信息到 `01_Shared/Strategy/`（用 Obsidian）
3. 然后让 Strategy Coach 基于 Vault 中的公开信息分析

### Q4：三个Claude Projects需要分开创建吗？

**A**：推荐分开创建，但可以根据实际情况调整：

- **理想方案**：3个独立Projects，各自有专门的System Prompt
- **简化方案**：1个Project，通过对话指令切换角色
- **Claude Code方案**：直接在本地使用，通过读取不同context文件切换

### Q5：规则库和Atlas/Dots/Skills有什么区别？

**A**：

| 维度 | Rules（规则库） | Atlas/Dots/Skills |
|-----|---------------|-------------------|
| **内容** | 什么时候该做什么（决策指南） | 如何做某件事（方法技能） |
| **触发** | 条件触发（如：涉及时间时...） | 主动调用（如：需要复盘时...） |
| **示例** | "涉及金额超过100元时提醒确认" | "复盘框架：GRAI四步法" |

### Q6：如何处理Ideaverse原有的Prompts？

**A**：可以保留，作为思维工具的补充：

- **保留有价值的**：Finding Blindspots、Imaginary Advisors
- **放在**：`x/Prompts/Ideaverse/` 子目录
- **使用场景**：在战略对话中引入多元视角

### Q7：这个系统会不会太复杂？

**A**：关键是**渐进式采用**：

1. **第一阶段**：只用 00_Core + 01_Shared/Calendar + 01_Shared/Strategy
2. **第二阶段**：加入 Efforts/Projects + 周复盘
3. **第三阶段**：完善 Atlas + 三个Claude身份
4. **第四阶段**：启用 02_Human + 03_AI 边界

**核心原则**：先用起来，再逐步完善。

### Q8：我们的 Skills 与 Anthropic Agent Skills 有什么关系？

**A**：两者理念相同，但定位不同：

**Anthropic Agent Skills**（2025年12月发布）：
- **定位**：AI Coding 工具的开放标准
- **核心**：渐进式披露（元数据 → 指令 → 资源）
- **格式**：`.claude/skills/` + YAML frontmatter
- **场景**：Claude Code、Cursor、VS Code 等工具

**我们的 Skills**：
- **定位**：个人知识管理系统中的可复用技能
- **核心**：索引优先，按需加载（与 Anthropic 理念一致）
- **格式**：`Atlas/Dots/Skills/` + Obsidian 双链
- **场景**：个人成长的复利积累 + AI 协作

**兼容性**：
- ✅ 可以轻松转换：我们的 Skills → Anthropic 标准格式
- ✅ 可以双向同步：在 Claude Code 中创建的技能 → 导入回 Obsidian
- ✅ 最佳实践：在 Obsidian 中管理（利用双链、MOC），需要时导出为标准格式

**核心差异**：
| 维度 | Anthropic Agent Skills | 我们的 Skills |
|-----|----------------------|-------------|
| 存储 | 独立目录 `.claude/skills/` | 融入知识库 `Atlas/Dots/Skills/` |
| 关联 | 技能相对独立 | 通过 MOC、双链深度关联 |
| 生命周期 | 开发者创建，AI执行 | AI使用中自动沉淀，持续迭代 |

**简单来说**：Anthropic Agent Skills 是"工具标准"，我们的 Skills 是"知识管理"，两者可以互补使用。

AI数字分身系统的三区框架
├─ 定义清晰的人机边界（00_Core / 02_Human / 03_AI）

├─ 项目层推进落地（Efforts/）
├─ 能力层持续积累（Atlas/）
└─ Ideaverse 知识管理框架（MOC、双链、模板）
        ↓
      融合后
├─ ✅ 01_Shared/ = PDCA 系统 + Ideaverse 知识管理（Obsidian）
├─ ✅ 02_Human/ = 私人空间（工具任选）
├─ ✅ 工具灵活选择，推荐而非强制
├─ ✅ 控制权始终在人手中
└─ ✅ AI是强力辅助，而非主人
